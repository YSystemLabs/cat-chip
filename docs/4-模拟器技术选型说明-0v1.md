# 周期级模拟器技术选型说明（v0.1）

> 版本：v0.1
>
> 状态：使用中

## 0. 文档目的

本文档用于回答一个具体工程问题：

> 对于当前第一阶段的 BlockMat 执行模型，软件侧应如何实现一个周期级模拟器？使用什么语言、什么架构、如何组织模块，才能既服务原型验证，又为后续硬件实现保留迁移路径？

本文档的目标不是定义新的执行语义，而是在既有规格 [技术规格说明-0v2.md](技术规格说明-0v2.md) 的基础上，给出一份可落地的技术选型建议。

本文档的适用范围是：

1. 第一阶段原型实现。
2. 周期级模拟器与代价报告生成。
3. BlockMat → SchedGraph → Schedule → CostReport 的软件闭环。

本文档不覆盖：

1. RTL 实现。
2. 物理设计与流片流程。
3. 通用 dense engine 的产品级性能对比。

---

## 1. 为什么需要周期级模拟器

当前项目的第一阶段目标不是只验证数值正确性，还要验证以下几点：

1. 结构模块与局部核是否真的被分离执行。
2. 广播、归约、核执行是否表现出独立资源约束。
3. 降低后的调度图能否导出可解释的周期数与资源轨迹。
4. 代价模型中的字段是否能与实际调度结果相互印证。

仅使用功能级解释器无法回答这些问题。

功能级解释器只能说明：

1. `eval(BlockMat)` 与 oracle 是否一致。
2. `compose`、`pair`、`copair` 等构造函数的语义是否正确。
3. R1-R8 是否保持数值语义不变。

但它无法说明：

1. 哪些周期在执行广播。
2. 哪些周期在执行核。
3. 归约端口何时冲突。
4. 实际调度周期与第 6 章上界公式相差多少。

因此，第一阶段至少需要一个**周期级软件模拟器**。

这里的“周期级”含义是：

1. 存在全局周期计数器 `cycle`。
2. 节点有显式开始与完成周期。
3. 资源在时间上占用与释放。
4. ready 节点按依赖与资源上限发射。

这与当前 spec 中的 `SchedGraph`、`scheduleGraph`、`CostReport` 是一致的。

## 2. 技术目标

一个合格的第一阶段周期级模拟器，至少要满足以下目标：

1. **语义正确**：执行结果与 `eval()` / oracle 一致。
2. **结构可观测**：广播、归约、直通、ZeroFill 等节点在 trace 中可见。
3. **资源可约束**：`P`、`B_ports`、`R_ports` 等资源参数真实参与调度。
4. **报告可对照**：能输出与 spec 第 6 章同口径的 `CostReport`。
5. **实现可扩展**：后续能增加更复杂的仲裁、优先级、双缓冲和统计项，而无需推翻当前架构。

由此可以反推语言与架构需求：

1. 需要清晰表达 IR、调度图、状态机和资源表。
2. 需要良好的测试支持。
3. 需要能导出 trace、report、JSON 等可分析格式。
4. 不要求一开始就逼近 RTL 细节。

### 2.1 数值口径约定

本选型文档沿用规格 [技术规格说明-0v2.md](技术规格说明-0v2.md) 的双通路口径：

1. **正确性通路**：第一版统一使用 `f32` 实现 `eval()`、oracle 路径与模拟器数值执行。
2. **性能通路**：FP16 不要求在第一版中做真实数值运算；其作用主要体现在代价模型参数中，例如 $W = 2$ 、`B_w`、`W_out` 等常量。
3. **误差准则**：端到端对照仍按规格中的 FP32 口径进行，即相对误差阈值保持在 $10^{-6}$ 量级。

这意味着第一版 Rust 原型不需要引入真实 `f16` 运算类型；若后续要评估混合精度数值误差，再单独扩展即可。

## 3. 语言选型比较

### 3.1 Python

优点：

1. 开发速度最快。
2. 数值计算库成熟，适合实现 `eval()` 和 oracle。
3. 非常适合做第一版语义验证器与快速实验。

缺点：

1. 大量状态对象和资源表在工程化阶段容易失控。
2. 随着模块增多，类型约束弱，重构成本上升。
3. 周期级模拟器后期会出现“功能都能跑，但状态一致性靠人盯”的问题。

适合：

1. 快速概念验证。
2. 语义解释器。
3. 作为 Rust/C++ 实现的参考 oracle。

### 3.2 C++

优点：

1. 性能高。
2. 传统体系结构模拟器常用，经验丰富。
3. 更靠近硬件/系统工具链生态。

缺点：

1. 第一版原型开发成本偏高。
2. 数据结构与状态机代码容易偏重工程细节。
3. 对当前阶段而言，投入产出比不如 Rust。

适合：

1. 大规模性能模拟。
2. 长期体系结构平台。
3. 已有 C++ 基础设施的团队。

### 3.3 Rust

优点：

1. 很适合表达 IR、调度图、资源状态、端口占用等强结构对象。
2. 类型系统能约束非法状态，降低周期级模拟中的隐式错误。
3. 模块化、测试、序列化、CLI 工具链都较成熟。
4. 相比 Python，更容易把原型演化成长期维护的工具。

缺点：

1. 第一版编码速度低于 Python。
2. 对团队要求更高，需要接受更严格的数据建模。
3. 若数学原型仍剧烈变动，Rust 的重构成本会高于 Python。

适合：

1. 当前这种对象模型已经基本稳定的项目。
2. 需要周期级模拟、trace、统计、可维护工具链的场景。
3. 希望后续与硬件实现或更强工程化平台对接的项目。

### 3.4 结论

结合当前项目状态，推荐结论如下：

1. **主实现语言推荐 Rust**。
2. **功能 oracle 可选 Python 保留一份轻量参考实现**。

原因不是 Rust “更高级”，而是当前 spec 已经基本稳定，项目重点已经从“解释概念”转向“把 IR、调度图和资源模型写成可靠原型”。在这个阶段，Rust 的收益开始明显大于 Python 的开发速度优势。

## 4. 推荐总体架构

推荐采用五层架构：

```text
IR/语义层
  -> 优化层
  -> Lowering 层
  -> 调度/模拟层
  -> 报告/验证层
```

### 4.1 IR/语义层

职责：

1. 定义 `KernelRef`、`BlockMat`、`Morph`。
2. 实现 `kernelAdd`、`kernelCompose`、`compose`。
3. 实现 `proj`、`inj`、`pair`、`copair`、`bimap` 等构造函数。
4. 实现 `eval()`。

输出：

1. 数值正确性。
2. 结构等式测试的基础语义。

### 4.2 优化层

职责：

1. 实现 R1-R8。
2. 做 kernel 分类：Zero / Id / Scalar / General。
3. 收集结构统计：`N_nz`、`N_gen`、`N_scale`、`N_unique`、`c_j`、`r_i`。

输出：

1. 优化后的 BlockMat。
2. 结构摘要。
3. 供代价模型使用的统计字段。

### 4.3 Lowering 层

职责：

1. 把优化后的 BlockMat 转为 `SchedGraph`。
2. 显式生成 `BroadcastNode`、`CoreNode`、`DirectNode`、`ReduceNode`、`ZeroFillNode`。
3. 建立稳定 NodeId 与 dependencies。

输出：

1. 可拓扑排序的 DAG。
2. 与硬件模块对应的显式执行对象。

### 4.4 调度/模拟层

职责：

1. 读取 `SchedGraph` 和 `SchedParams`。
2. 维护 ready 队列、已发射集合、完成周期表。
3. 按 `coreSlots`、`bcastPorts`、`reducePorts` 进行周期推进。
4. 输出 `Schedule`、资源轨迹和总周期。

输出：

1. `timedNodes`。
2. `issuedBatches`。
3. 实际周期数。

### 4.5 报告/验证层

职责：

1. 从 `Morph`、`SchedGraph`、`Schedule` 汇总 `CostReport`。
2. 对照第 6 章解析公式与实际模拟结果。
3. 对照 oracle 数值结果。
4. 输出 JSON、表格或 trace。

输出：

1. 正确性报告。
2. 周期报告。
3. 结构收益报告。

## 5. Rust 推荐模块划分

建议以一个 workspace 或单 crate 多模块方式组织。

### 5.1 模块结构

```text
src/
  lib.rs
  ir.rs
  semantics.rs
  optimize.rs
  lower.rs
  schedule.rs
  simulate.rs
  report.rs
  oracle.rs
  trace.rs
  tests/
```

### 5.2 各模块职责

`ir.rs`

1. 定义 `KernelRef`、`BlockMat`、`Morph`。
2. 定义验证错误类型。

`semantics.rs`

1. 实现 `compose`、`eval`。
2. 实现结构构造函数。

`optimize.rs`

1. 实现 R1-R8。
2. 实现 kernel 分类与统计。

`lower.rs`

1. 定义 `NodeId`、`NodeType`、`SchedNode`、`SchedGraph`。
2. 实现 `lower()`。

`schedule.rs`

1. 定义 `SchedParams`、`Batch`、`TimedNode`、`Schedule`。
2. 实现 `topo_sort`、`ready_nodes`、`issue_batch`、`schedule_graph`。
3. 负责**纯时序调度**：只决定节点何时可发射、何时完成、资源何时被占用，不负责具体数值计算。

`simulate.rs`

1. 在 `Schedule` 给出的时间表上推进**数值状态**与**缓冲区状态**。
2. 根据节点类型执行广播、局部核、归约、ZeroFill 对数据的作用。
3. 记录每周期资源占用、节点完成、缓冲区峰值占用等运行轨迹。

换言之，`schedule.rs` 是“谁在什么时候运行”的时序调度器，`simulate.rs` 是“这些节点在既定时序下对数据和缓冲区做了什么”的周期执行器。

`report.rs`

1. 定义 `CostReport`。
2. 实现 `report_from_schedule()`。
3. 按规格同时输出 $C_{\mathrm{load}}$ 与 $C_{\mathrm{load}}^{\mathrm{eff}}$ 。

`oracle.rs`

1. 不重复实现 `eval()` 本身。
2. 负责端到端对照 harness：接收 `(BlockMat, 输入向量, Schedule/SimResult)`，分别走功能语义路径与模拟器路径，并比较误差是否低于阈值。

`trace.rs`

1. 导出 JSON trace。
2. 生成人类可读报告。

## 6. 核心数据结构建议

### 6.1 IR 层

建议用显式枚举与 struct，而不是动态字典。

这里需要明确区分**原始 IR 层**与**优化后表示层**。

原始 IR 层应保持与规格一致，仅保留三种 kernel 变体：`General(Mat2)` / `Id` / `Zero`。标量核在数学上仍属于 `General(c·I)`，不应在原始 IR 层提前出现。

R3 之后若希望提高工程实现清晰度，可以在优化结果中引入单独的标量变体；此时建议使用不同的枚举类型，而不是直接扩展原始 IR：

1. `KernelRef`：原始 IR 使用，保持与 spec 一致。
2. `LoweredKernelRef` 或 `ExecKernelRef`：优化后 / 执行层使用，可包含 `Scalar(f32)`。

这样可以避免把优化标记倒灌回数学 IR，本体代数闭合性也更容易维护。

```rust
pub enum KernelRef {
    General(Mat2),
    Id,
    Zero,
}

pub struct BlockMat {
    pub rows: usize,
    pub cols: usize,
    pub grid: Vec<Vec<KernelRef>>,
}
```

若要显式表示标量核，推荐在优化后层单独定义：

```rust
pub enum ExecKernelRef {
    General(Mat2),
    Id,
    Zero,
    Scalar(f32),
}
```

其中 `Scalar(f32)` 仅在 R3 识别后出现，用于调度与代价计算，不改变原始 IR 定义。

### 6.2 调度图层

```rust
pub type NodeId = usize;

pub enum NodeType {
    Broadcast,
    Core(ExecKernelRef),
    Direct,
    Reduce,
    ZeroFill,
}

pub struct SchedNode {
    pub id: NodeId,
    pub node_type: NodeType,
    pub input_slots: Vec<BufferAddr>,
    pub output_slots: Vec<BufferAddr>,
    pub param_handle: Option<ParamAddr>,
    pub deps: Vec<NodeId>,
}
```

这里 `Core` 应显式携带 `ExecKernelRef` 而不是原始 IR 层的 `KernelRef`。原因是 lowering 发生在 R3 之后，调度图需要保留“该核是否已被识别为标量核”的信息，否则调度器无法区分 `L_core` 与 `L_scale`。

### 6.3 资源状态层

建议显式建模资源占用状态：

```rust
pub enum SlotState {
    Idle,
    Busy { node: NodeId, until: usize },
}

pub struct MachineState {
    pub cycle: usize,
    pub core_slots: Vec<SlotState>,
    pub bcast_ports: Vec<SlotState>,
    pub reduce_ports: Vec<SlotState>,
    pub input_buffer_occupancy: usize,
    pub temp_buffer_occupancy: usize,
    pub output_buffer_occupancy: usize,
    pub peak_temp_buffer_occupancy: usize,
}
```

这类建模与很多 Rust 周期级模拟器的常见写法一致，也最容易调试。这里特别建议第一版就至少跟踪：

1. 临时缓冲峰值占用。
2. 输入块释放时刻。
3. 输出块写回完成时刻。

原因是规格已经对共享缓冲区容量和生命周期给出了明确约束，因此模拟器至少要能验证“是否超出容量上界”。

缓冲区追踪单位建议声明为：**内部按块数追踪，报告层按规格口径换算为标量数**。也就是：

1. `input_buffer_occupancy`、`temp_buffer_occupancy`、`output_buffer_occupancy` 在模拟器内部表示“当前占用多少个块槽位”。
2. 在 `report.rs` 中再乘以块宽 `d`，映射为规格 §7.4 使用的“标量数”口径。

这样内部状态与调度图粒度一致，而外部报告仍与 spec 主文保持一致。

### 6.4 调度参数集

为与规格第 6 章直接对齐，推荐 `SchedParams` 至少包含以下字段：

```rust
pub enum LatencyModel {
    Tree,        // ceil(log2(k))
    Pipeline,    // k
    Fixed(usize),
}

pub struct SchedParams {
    pub core_slots: usize,     // P
    pub bcast_ports: usize,    // B_ports
    pub reduce_ports: usize,   // R_ports
    pub core_latency: usize,   // L_core
    pub scale_latency: usize,  // L_scale
    pub direct_latency: usize, // 直通延迟，第一版可取 0 或 1
    pub bw_params: usize,      // B_w，参数总线宽度（标量/周期）
    pub bw_writeback: usize,   // W_out，输出写端口带宽（标量/周期）
    pub bcast_model: LatencyModel,
    pub reduce_model: LatencyModel,
}

pub fn latency(model: &LatencyModel, k: usize) -> usize {
    match model {
        LatencyModel::Tree => ceil_log2(k),
        LatencyModel::Pipeline => k,
        LatencyModel::Fixed(v) => *v,
    }
}
```

其中：

1. `latency(&bcast_model, k)` 对应规格中的 $C_{\mathrm{bcast},j}$ 。
2. `latency(&reduce_model, k)` 对应单行归约延迟 $C_{\mathrm{reduce},i}$ 。
3. `bw_params` 和 `bw_writeback` 直接用于解析报告中的 $C_{\mathrm{load}}$ 与 $C_{\mathrm{wb}}$ 。

这样 `schedule.rs`、`simulate.rs` 与 `report.rs` 可以共享同一组参数，不必各自重建口径。采用枚举策略而不是函数指针的好处在于：参数对象更容易序列化、打印、比较和写入 trace / JSON 配置。

### 6.5 模拟结果类型

为使 `simulate.rs`、`report.rs` 与 `oracle.rs` 的接口闭合，建议显式给出最小 `SimResult` 草案：

```rust
pub struct SimResult {
    pub output_vectors: Vec<Vec<f32>>,       // n 个 d 维输出块
    pub peak_temp_buffer: usize,             // 临时缓冲峰值（块数）
    pub per_cycle_trace: Vec<CycleSnapshot>, // 可选：逐周期轨迹
}
```

其中：

1. `output_vectors` 用于和 `eval()` / oracle 做数值对照。
2. `peak_temp_buffer` 用于验证共享缓冲区容量约束。
3. `per_cycle_trace` 供 `trace.rs` 和调试使用，第一版可选实现。

这一定义不要求立刻把所有字段做全，但至少让文档层的模块接口是闭合的。

## 7. 推荐实现策略

### 7.1 第一阶段的正确路线

建议实现顺序如下：

1. 先完成 IR + eval + 构造函数。
2. 再完成 R1-R8 与结构统计。
3. 再完成 lowering 到 `SchedGraph`。
4. 再完成最小调度器。
5. 最后补 trace、report 和 JSON 导出。

不要一上来写完整模拟器主循环，否则很容易在没有稳定 oracle 的前提下同时调试语义错误与调度错误。

### 7.2 第一版不要做的事

第一版不建议做：

1. 复杂事件队列。
2. 多级缓存模型。
3. 动态优先级仲裁。
4. 波形级仿真接口。
5. 与 RTL 联动。

原因是这些复杂度不会帮助第一阶段证明方法论，反而会拖慢闭环。

### 7.3 第一版应该保留的扩展点

第一版建议预留但不深入实现：

1. 资源类枚举。
2. trace 导出格式。
3. 不同发射策略的 trait 或策略对象。

这样后续可以从“最小原型”平滑升级到“更接近微架构”的模拟器。

### 7.4 第一版必须实现的口径项

以下内容不应再视为“未来扩展”，因为规格主文已经把它们纳入第一阶段验收口径：

1. **双缓冲装载的报告字段**：即使调度器本身不显式模拟参数双缓冲，也必须在 `report.rs` 中按规格公式输出 $C_{\mathrm{load}}$ 和 $C_{\mathrm{load}}^{\mathrm{eff}}$ 两个值。
2. **结构统计字段**：`N_nz`、`N_gen`、`N_scale`、`N_unique`、`c_j`、`r_i` 必须进入报告。
3. **解析值与实测值并存**：既要有按第 6 章计算的解析上界，也要有 `schedule_graph()` / `simulate.rs` 导出的实际周期数。

这里的关键点是：双缓冲可以先作为**报告层建模项**实现，而不要求第一版调度器先做复杂的动态 overlap 仿真。

## 8. 与 ZK/RISC-V 周期级模拟器经验的关系

Rust 编写的 RISC-V / ZK 周期级模拟器可借鉴之处，主要在于：强类型状态机、显式资源占用、trace 驱动验证、以及“先做周期级原型、再逼近 RTL”的工程路径。

但它们的核心对象通常是 instruction / pipeline / register-file，而当前项目的核心对象是 graph node / block dataflow / resource graph。因此，可借鉴的是工程组织方法，不是其执行对象模型本身。

---

## 9. 最终推荐

综合建议如下：

1. **主实现语言：Rust**。
2. **数值 oracle：可选保留 Python 或 Rust 中的纯功能路径**。
3. **架构风格：IR/优化/lower/schedule/report 五层分离**。
4. **调度风格：先做最小周期推进器，再逐步增强，不直接追求 RTL 级精度**。

如果只用一句话概括这份选型建议，那就是：

> **对当前第一阶段项目，最合理的做法不是先做“像硬件一样复杂”的模拟器，而是用 Rust 做一个强类型、可测试、可追踪的周期级原型，把 BlockMat 执行模型完整跑通。**

## 10. 实施入口

如果按本文档启动实现，推荐第一批交付物如下：

1. `BlockMat` / `KernelRef` / `eval()`。

3. `SchedGraph` 与 `lower()`。

完成这七项后，第一阶段就已经具备一个真正意义上的软件原型闭环。
