# 技术规格说明（v0.2）—— 双积分解与块矩阵执行模型

> 版本：v0.2
>
> 状态：已实施

## 0. 文档定位

本文档是第一阶段的技术规格说明。

第一阶段的作用域严格限定为：

- **对象**：固定块 $B = k^d$（$d = 2$）的有限直和 $B^{\oplus n}$
- **态射**：块矩阵 $\mathrm{Mat}_{n \times m}(\mathrm{End}(B))$，即 $n \times m$ 网格的参数化 $d \times d$ 线性核
- **不涉及**：张量积（$\otimes$）侧的硬件原语、Kronecker 置换、Systolic Array、分配律同构

本文档要回答的问题是：

> **当态射被表示为块矩阵时，IR 如何定义、编译器如何优化、硬件如何执行、如何验证正确性？**

前置文档：《[执行模型说明-0v2](2-执行模型说明-0v2.md)》。本文档的所有设计决策均与执行模型对齐。

> **实施边界说明（重要）**
>
> 本文档作为第一阶段“可表示、可编译、可调度、可验证”的**原型 spec**，已经足够支撑软件原型与周期级模拟器实现。
>
> 但它**不是**可直接投产的硬件微架构规格书。若要进入 RTL/流片导向的实现阶段，仍需在本 spec 之上补充至少以下几层：
>
> 1. **接口层**：共享缓冲地址编码、参数句柄格式、控制寄存器布局、调度表编码。
> 2. **仲裁层**：广播端口冲突处理、归约端口分配策略、写回端口仲裁、背压传播规则。
> 3. **时序层**：各结构模块的逐周期握手协议、流水线级间寄存、旁路路径时序约束。
> 4. **实现层**：面积/功耗模型、SRAM 组织、参数装载路径、时钟频率目标与物理约束。
>
> 因此，本文档的定位应理解为：**它已经足够用于第一阶段原型实施；但距离直接投产的硬件微架构 spec，还差若干实现层规格。**

### 0.1 数值口径约定

本文档的数值假设分为两条独立通路，职责不同，不混用：

| 通路 | 数值格式 | 用途 | 字宽 $W$ |
|------|---------|------|----------|
| **正确性通路** | FP32 | oracle 求值器、BlockMat `eval()`、执行器-oracle 数值对照 | 4 字节 |
| **性能通路** | FP16 | 代价模型、算术强度 $I(d)$、缓冲容量估算、代价比 $R$ | 2 字节 |

- 正确性通路中的等式验证使用相对误差 $< 10^{-6}$（FP32 精度）。
- 性能通路中的所有带宽、延迟和面积估算统一基于 $W = 2$ 字节（FP16）。
- 微架构的核阵列物理实现以 FP32 为验证格式、FP16 为目标格式。验收条件中的数值正确性以 FP32 oracle 为基准。
- **第一阶段模拟器仅实现 FP32 正确性通路**。模拟器中的所有数值计算（oracle `eval()`、`simulate()` 执行）均使用 FP32。FP16 数值模拟（包括精度损失、截断行为等）在后续阶段补充；性能通路当前仅用于代价报告中的带宽和容量估算。

## 1. 设计原则

### 1.1 单一 IR + 结构构造子

系统采用**单一 IR**设计——`BlockMat(n, m, K)` 是唯一的中间表示。所有从 $B^{\oplus m}$ 到 $B^{\oplus n}$ 的线性映射统一表示为 $n \times m$ 的块矩阵。优化器（R1-R8）、代价模型、调度器和验证器均直接消费 BlockMat。

结构操作（投影、注入、配对、余配对、复制、合并等）通过一组**类型化构造函数**（§3.5）实现，直接产出 BlockMat。这些构造函数是具名的、带类型签名的、语义明确的——满足宣言对结构操作显式可追踪的要求。

**结构在构造过程中显式，在产出的 IR 中编码为块矩阵模式。** 结构操作不是被"跳过"，而是被系统性地映射为确定的块矩阵模式（§3.3）。这些模式在 R1-R8 优化阶段被识别和利用。

块矩阵的完备表示能力由 §3.4 的覆盖性定理保证——有限直和子范畴中的任意态射都能被块矩阵精确表示。这使得单一 IR 设计在数学上无信息丢失。

### 1.2 Eager Flattening

两个块矩阵的 `Compose` 在 IR 中**立即展开**为一个新的块矩阵（通过块矩阵乘法规则），不保留符号化的 Compose 节点（对齐执行模型 §3.3.3 Eager Flattening 策略）。

理由：第一阶段验证目标是"单个态射的端到端闭环"，不需要多级管道表示。这使 IR 在任何时刻都是一个单一 BlockMat，极大简化了下游处理。

### 1.3 结构操作：构造显式，IR 编码

结构操作通过类型化构造函数（§3.5）引入，其类型签名和语义如下：

| 构造函数 | 类型签名 | 含义 |
|---------|---------|------|
| `proj(i, n)` | $B^{\oplus n} \to B$ | 取第 $i$ 个块 |
| `inj(i, n)` | $B \to B^{\oplus n}$ | 嵌入第 $i$ 个位置 |
| `pair(f, g)` | $A \to B^{\oplus (p+q)}$ | 配对（纵向拼接） |
| `copair(f, g)` | $B^{\oplus (p+q)} \to C$ | 余配对（横向拼接） |
| `copy(n)` | $B \to B^{\oplus n}$ | 复制（对角态射） |
| `merge(n)` | $B^{\oplus n} \to B$ | 合并（余对角态射，加法归约） |

每个构造函数直接产出 BlockMat，结构操作被编码为确定的块矩阵模式：

| 结构操作 | 产出的块矩阵模式 |
|---------|----------------|
| `proj(i, n)` — 取第 $i$ 个输入块 | 单行块矩阵，仅第 $i$ 列为 IdKernel |
| `inj(i, n)` — 写第 $i$ 个输出块 | 单列块矩阵，仅第 $i$ 行为 IdKernel |
| `copy(n)` — 一个输入块被多路读取 | 块矩阵某一列有多个非零块 |
| `merge(n)` — 多路输出归约到同一块 | $y_i = \sum_j K[i][j](x_j)$，即每行的归约求和 |
| `discard` — 某输入块不被消费 | 块矩阵某一列全为 ZeroKernel |

结构操作在 BlockMat 中被编码为核引用模式，**在 lowering 到调度图时再次显式化**。

这一设计的意义：
- 宣言要求的"结构显式"在构造函数的具名调用中兑现——每个结构操作有明确的名称、类型签名和语义，编译器在 IR 上可通过模式识别还原并利用这些结构；
- 执行模型要求的"块矩阵作为编译目标"由 BlockMat 作为唯一 IR 直接兑现；
- 结构操作到块矩阵模式的映射由 §3.4 覆盖性定理保证无损。

### 1.4 代价敏感

结构合法不等于物理高效。每个块矩阵的调度方案同时携带**结构描述**（哪些核需要执行、哪些可消去）和**代价标注**（延迟、带宽、归约开销）。编译器的优化目标不是"减少 IR 节点数"，而是"在等价块矩阵中选择代价最低的调度方案"。

### 1.5 最小闭环优先

不追求完备性或最优性，只追求 **IR → 优化 → 调度图 → 执行 → 验证** 的闭环可运行性。

## 2. 对象类型系统

### 2.1 基本对象类型

第一阶段的对象类型由以下文法生成：

```
Type ::= Block           -- 基础块 B = k^d，第一阶段 d = 2
       | Blocks(n)       -- B^{⊕n}，n 个基础块的直和（n ∈ ℕ⁺）
```

**约束**：

- `Block` 等价于 `Blocks(1)`。
- 所有类型的维度在编译时完全确定（无动态维度）。
- **不允许嵌套**：不存在 `Blocks(Blocks(...))`。所有对象都是扁平的 $B^{\oplus n}$。
- **第一阶段不引入零对象**：`discard` 语义通过块矩阵中的零列模式（R5 死列消去）实现，不需要独立的 Void 类型。

### 2.2 维度计算

每个类型 $T$ 关联一个编译时常量 $\dim(T)$：

$$
\begin{aligned}
\dim(\mathtt{Block}) &= d \quad (\text{第一阶段 } d = 2) \\
\dim(\mathtt{Blocks}(n)) &= n \cdot d
\end{aligned}
$$

### 2.3 对象域的覆盖边界

第一阶段只接受维度为 $d$ 的倍数的对象。对于不能被 $d$ 整除的维度：

- **奇数维输入**：前端预处理阶段补零嵌入到最近的 $d$ 的倍数。当 $d = 2$ 时，$N$ 维对象嵌入到 $N+1$ 维。补零引入的新列/行在块矩阵中均为 `ZeroKernel`，可被编译器消去。
- 当 $N \gg d$ 时，补零开销占比可忽略。

## 3. 原语与生成态射

### 3.1 唯一的计算原语：参数化局部核

第一阶段的计算原语只有一种：

$$
\mathtt{kernel}(M) : B \to B \qquad M \in k^{d \times d}
$$

所有从 $B$ 到 $B$ 的线性映射都由一个 $d \times d$ 参数矩阵 $M$ 完全确定。

两个重要特例在编译器中有专门处理：

| 核引用 | 参数 | 语义 | 执行代价 |
|--------|------|------|---------|
| `Kernel(M)` | 一般 $d \times d$ 矩阵 $M$ | $v \mapsto Mv$ | $L_{\text{core}}$ 周期 |
| `IdKernel` | $M = I_d$ | $v \mapsto v$（恒等） | 0（直通连线） |
| `ZeroKernel` | $M = 0$ | $v \mapsto 0$（零映射） | 0（不执行，不调度） |

### 3.2 从块到直和：块矩阵作为唯一态射构造

从 $B^{\oplus m}$ 到 $B^{\oplus n}$ 的态射统一表示为一个 $n \times m$ 的块矩阵：

$$
\mathtt{BlockMat}(n, m, K) : \mathtt{Blocks}(m) \to \mathtt{Blocks}(n)
$$

其中 $K$ 是一个 $n \times m$ 的二维数组，$K[i][j]$ 是一个 `KernelRef`（`Kernel(M)` | `IdKernel` | `ZeroKernel`）。

块矩阵是态射的**唯一表示形式**。双积结构操作不出现在这一层。

### 3.3 结构操作作为块矩阵的特殊模式

以下结构态射无需独立原语，它们是块矩阵的特殊实例：

| 结构态射 | 块矩阵表示 | 说明 |
|---------|-----------|------|
| $\text{id}_{B^{\oplus n}}$ | `BlockMat(n, n, K)` 其中 $K[i][j] = \delta_{ij} \cdot \mathtt{IdKernel}$ | 对角恒等矩阵 |
| $\pi_i : B^{\oplus n} \to B$ | `BlockMat(1, n, K)` 其中 $K[0][j] = \delta_{ij} \cdot \mathtt{IdKernel}$ | 单行，仅第 $i$ 列为 IdKernel |
| $\iota_i : B \to B^{\oplus n}$ | `BlockMat(n, 1, K)` 其中 $K[j][0] = \delta_{ij} \cdot \mathtt{IdKernel}$ | 单列，仅第 $i$ 行为 IdKernel |
| $\Delta_B : B \to B \oplus B$ | `BlockMat(2, 1, [[IdKernel], [IdKernel]])` | 两行一列，均为 IdKernel |
| $\nabla_B : B \oplus B \to B$ | `BlockMat(1, 2, [[IdKernel, IdKernel]])` | 一行两列，均为 IdKernel（归约为加法） |
| $\sigma^\oplus : B^{\oplus m} \oplus B^{\oplus n} \to B^{\oplus n} \oplus B^{\oplus m}$ | 相应的置换块矩阵 | 对角位置上的 IdKernel 按置换重排 |
| $f \oplus g$（块对角） | $K$ 为块对角形式 | $f$ 占据左上块，$g$ 占据右下块，其余为 ZeroKernel |

这一设计的核心洞察：**双积范畴中的全部结构态射，在固定块的直和子范畴中，都退化为块矩阵的特殊形状。** 独立原语集为空——块矩阵本身就是唯一的生成构造。

### 3.4 态射覆盖性

**定理**（执行模型 §3）：在由 $B = k^d$ 生成的有限直和子范畴中，

$$
\mathrm{Hom}(B^{\oplus m}, B^{\oplus n}) \cong \mathrm{Mat}_{n \times m}(\mathrm{End}(B)) \cong \mathrm{Mat}_{n \times m}(\mathrm{Mat}_d(k))
$$

因此 `BlockMat(n, m, K)` 对该子范畴中的态射具有完备表示能力。

### 3.5 结构构造函数

本节定义一组类型化构造函数，将双积结构操作直接映射为 BlockMat。这些函数是使用者构造结构化程序的接口——每次调用都产出一个合法的 `BlockMat(n, m, K)`，不引入独立的中间表示。

#### 3.5.1 基本构造函数

| 构造函数 | 签名 | 产出 |
|---------|------|------|
| `kernelApp(M)` | $M \in k^{d \times d} \to \mathtt{Morph}$ | `BlockMat(1, 1, [[Kernel(M)]])` |
| `proj(i, n)` | $0 \leq i < n \to \mathtt{Morph}$ | `BlockMat(1, n, K)` 其中 $K[0][j] = \delta_{ij} \cdot \mathtt{IdKernel}$ |
| `inj(i, n)` | $0 \leq i < n \to \mathtt{Morph}$ | `BlockMat(n, 1, K)` 其中 $K[j][0] = \delta_{ij} \cdot \mathtt{IdKernel}$ |
| `copy(n)` | $n \geq 2 \to \mathtt{Morph}$ | `BlockMat(n, 1, [[IdKernel], ..., [IdKernel]])` |
| `merge(n)` | $n \geq 2 \to \mathtt{Morph}$ | `BlockMat(1, n, [[IdKernel, ..., IdKernel]])` |
| `identity(n)` | $n \geq 1 \to \mathtt{Morph}$ | 对角 IdKernel 的 $n \times n$ 块矩阵 |

#### 3.5.2 组合构造函数

| 构造函数 | 签名 | 产出 |
|---------|------|------|
| `pair(f, g)` | $f : B^{\oplus m} \to B^{\oplus p}$, $g : B^{\oplus m} \to B^{\oplus q}$ $\to \mathtt{Morph}$ | 将 $f$ 和 $g$ 的核网格纵向拼接，产出 `BlockMat(p+q, m, ...)` |
| `copair(f, g)` | $f : B^{\oplus p} \to B^{\oplus n}$, $g : B^{\oplus q} \to B^{\oplus n}$ $\to \mathtt{Morph}$ | 将 $f$ 和 $g$ 的核网格横向拼接，产出 `BlockMat(n, p+q, ...)` |
| `bimap(fs)` | $[f_1, \ldots, f_k]$ $\to \mathtt{Morph}$ | 将各 $f_i$ 排列为块对角 |
| `compose(g, f)` | $f : A \to C$, $g : C \to D$ $\to \mathtt{Morph}$ | 按 §4.3 的 eager flattening 规则展开 |

组合构造函数接受 `Morph`（即 BlockMat）作为输入，直接产出新的 `Morph`。不存在符号化的 Compose 或 Pair 节点——所有组合在调用时立即求值为单一 BlockMat。

工程实现还应暴露一个派生 helper `morphAdd(f, g)`（Rust API 可命名为 `morph_add`），用于显式表示平行态射的加法，但它**不是新的 IR 节点**，而是以下双积构造的缩写：

$$
f + g = \nabla_B \circ (f \oplus g) \circ \Delta_A
$$

其中对 $f, g : A \to B$，有：

$$
\Delta_A = \langle \mathrm{id}_A, \mathrm{id}_A \rangle, \qquad
\nabla_B = [\mathrm{id}_B, \mathrm{id}_B], \qquad
f \oplus g = \mathrm{bimap}([f, g])
$$

因此，工程 API 可以提供显式 helper 以方便测试和上层构造，但底层仍保持“唯一 IR = BlockMat”的设计约束。

在有限维向量空间语境下，`Hom(A, B)` 还天然带有底层域 $k$ 上的线性结构。因此工程实现也可以暴露派生态射数乘 helper `morphScale(c, f)`（Rust API 可命名为 `morph_scale`），其语义是在 `BlockMat` 的每个核上做逐块数乘。它同样不是新的 IR 节点，只是对块矩阵条目算术的显式封装。

#### 3.5.3 结构等式（验证性质）

以下等式由双积的泛性质保证，是宣言所要求的"结构等式驱动编译收益"的数学基础。在本设计中，这些等式**不作为编译 pass**，而作为**验证性质**——用测试证明构造函数和 `compose` 在 BlockMat 层正确实现了这些等式：

| 编号 | 等式 | 来源 | BlockMat 层验证方式 |
|------|------|------|-------------------|
| S1 | $\pi_i \circ \iota_i = \text{id}_B$ | 双积公理 | `compose(proj(i,n), inj(i,n))` 产出 $1 \times 1$ IdKernel |
| S2 | $\pi_i \circ \iota_j = 0$ 当 $i \neq j$ | 双积公理 | `compose(proj(i,n), inj(j,n))` 产出 $1 \times 1$ ZeroKernel |
| S3 | $\nabla \circ \Delta = n \cdot \text{id}_B$ | 导出 | `compose(merge(n), copy(n))` 产出 `BlockMat(1,1,[[Kernel(n·I_d)]])` |
| S4 | $\langle \pi_0, \ldots, \pi_{n-1} \rangle = \text{id}_{B^{\oplus n}}$ | 积 η 律 | `pair(proj(0,n), pair(proj(1,n), ...))` 产出对角 IdKernel |
| S5 | $[\iota_0, \ldots, \iota_{n-1}] = \text{id}_{B^{\oplus n}}$ | 余积 η 律 | `copair(inj(0,n), copair(inj(1,n), ...))` 产出对角 IdKernel |
| S6 | $(f_1 \oplus f_2) \circ (g_1 \oplus g_2) = (f_1 \circ g_1) \oplus (f_2 \circ g_2)$ | 块对角 | `compose(bimap([f1,f2]), bimap([g1,g2]))` 等价于 `bimap([compose(f1,g1), compose(f2,g2)])` |
| S7 | $(\iota_0 \circ \pi_0) + (\iota_1 \circ \pi_1) = \text{id}_{B^{\oplus 2}}$ | 双积完备性 | `morphAdd(compose(inj(0,2), proj(0,2)), compose(inj(1,2), proj(1,2)))` 等价于 `identity(2)` |

这些等式的成立由 `compose` 函数的 eager flattening 机制（§4.3）和 `kernelCompose`/`kernelAdd` 的算术规则自动保证。R1-R8 优化在这些等式之上进一步识别和利用块矩阵中的结构模式。

在工程测试中，还应额外锁定 `morphAdd` 的基本定律：

- 零元：`morphAdd(f, 0) = f = morphAdd(0, f)`
- 交换律：`morphAdd(f, g) = morphAdd(g, f)`
- 结合律：`morphAdd(morphAdd(f, g), h) = morphAdd(f, morphAdd(g, h))`
- 左右分配律：`k ∘ morphAdd(f, g) = morphAdd(k ∘ f, k ∘ g)`，`morphAdd(f, g) ∘ h = morphAdd(f ∘ h, g ∘ h)`
- 标量单位元与零元：`morphScale(1, f) = f`，`morphScale(0, f) = 0`
- 标量结合律：`morphScale(ab, f) = morphScale(a, morphScale(b, f))`
- 标量对态射加法的分配律：`morphScale(a, morphAdd(f, g)) = morphAdd(morphScale(a, f), morphScale(a, g))`
- 标量加法的分配律：`morphScale(a+b, f) = morphAdd(morphScale(a, f), morphScale(b, f))`

这些测试的作用不是新增语义，而是把“Hom 集带加法”这条双积范畴的工程可用性显式回归化，避免实现只保留内部核加法却丢失上层接口。

**无需独立重写引擎**：S1-S7 不是在独立 IR 层上做的符号重写，而是 `compose` + 构造函数的固有算术结果。

## 4. 块矩阵 IR

BlockMat 是系统的唯一中间表示。结构构造函数（§3.5）直接产出 BlockMat，优化器（§5）、代价模型（§6）和调度器（§7）均消费此表示。本节定义 BlockMat IR 的语法、类型规则、组合规则和语义。

### 4.1 IR 语法

```
-- ===== 类型 =====
Type ::= Blocks(n)     -- B^{⊕n}，n ≥ 1

-- ===== 核引用 =====
KernelRef ::= Kernel(M)     -- 参数化 d×d 矩阵 M ∈ k^{d×d}
            | IdKernel       -- 恒等核 I_d
            | ZeroKernel     -- 零核 0_{d×d}

-- ===== 态射项 =====
Morph ::= BlockMat(n, m, K)     -- K : KernelRef[n][m]
                                 -- 类型：Blocks(m) → Blocks(n)
```

**注意**：BlockMat IR 中**不存在** `Compose` 节点，也不存在独立的结构操作节点——结构操作通过 §3.5 的构造函数直接产出 BlockMat 模式。两个块矩阵的组合在进入 IR 时立即被 eager flattening 展开为一个新的 `BlockMat`（见 §4.3）。IR 在任何时刻都是一个单一的 `BlockMat`。

### 4.2 良类型规则

$$
\frac{n, m \in \mathbb{N}^+ \qquad K : \mathtt{KernelRef}[n][m]}{\mathtt{BlockMat}(n, m, K) : \mathtt{Blocks}(m) \to \mathtt{Blocks}(n)}
$$

每个 `KernelRef` 的类型固定为 $B \to B$，无需额外类型检查。只需检查 $n, m \geq 1$ 且 $K$ 的维度与 $n, m$ 一致。

### 4.3 块矩阵组合（Eager Flattening）

给定两个块矩阵

$$
F = \mathtt{BlockMat}(n, m, K_F) : B^{\oplus m} \to B^{\oplus n}
$$

$$
G = \mathtt{BlockMat}(p, n, K_G) : B^{\oplus n} \to B^{\oplus p}
$$

它们的组合 $G \circ F : B^{\oplus m} \to B^{\oplus p}$ 立即展开为：

$$
\mathtt{BlockMat}(p, m, K_{G \circ F})
$$

其中每个块定义为：

$$
K_{G \circ F}[i][k] = \bigoplus_{j=1}^{n} K_G[i][j] \cdot K_F[j][k]
$$

右侧涉及两个核级操作：

- **核组合**（$\cdot$）：两个 $d \times d$ 矩阵的乘法
- **核求和**（$\bigoplus$）：多个 $d \times d$ 矩阵的逐元素加法

规则：
- $\mathtt{ZeroKernel} \cdot X = X \cdot \mathtt{ZeroKernel} = \mathtt{ZeroKernel}$
- $\mathtt{IdKernel} \cdot X = X \cdot \mathtt{IdKernel} = X$
- $\mathtt{ZeroKernel} \oplus X = X$（零核是加法单位元）
- $\mathtt{Kernel}(M_1) \oplus \mathtt{Kernel}(M_2) = \mathtt{Kernel}(M_1 + M_2)$
- $\mathtt{Kernel}(M_1) \cdot \mathtt{Kernel}(M_2) = \mathtt{Kernel}(M_1 M_2)$

**注意**：Eager flattening 引入了核级矩阵加法。这是 IR 内部的编译期运算，不影响执行模型——执行层面只看到展开后的单一块矩阵。此处的核加法对应的正是双积完备性等式 $\iota_1 \circ \pi_1 + \iota_2 \circ \pi_2 = 1_{V \oplus W}$ 在块矩阵层面的体现。

### 4.4 求值语义

定义求值函数 $\lbrack\!\lbrack - \rbrack\!\rbrack$，将 IR 项映射到 $k^{nd \times md}$ 矩阵：

$$
\lbrack\!\lbrack \mathtt{BlockMat}(n, m, K) \rbrack\!\rbrack = \begin{pmatrix}
\lbrack\!\lbrack K[0][0] \rbrack\!\rbrack & \lbrack\!\lbrack K[0][1] \rbrack\!\rbrack & \cdots & \lbrack\!\lbrack K[0][m-1] \rbrack\!\rbrack \\
\lbrack\!\lbrack K[1][0] \rbrack\!\rbrack & \lbrack\!\lbrack K[1][1] \rbrack\!\rbrack & \cdots & \lbrack\!\lbrack K[1][m-1] \rbrack\!\rbrack \\
\vdots & \vdots & \ddots & \vdots \\
\lbrack\!\lbrack K[n-1][0] \rbrack\!\rbrack & \lbrack\!\lbrack K[n-1][1] \rbrack\!\rbrack & \cdots & \lbrack\!\lbrack K[n-1][m-1] \rbrack\!\rbrack
\end{pmatrix}
$$

其中：

$$
\begin{aligned}
\lbrack\!\lbrack \mathtt{Kernel}(M) \rbrack\!\rbrack &= M \in k^{d \times d} \\
\lbrack\!\lbrack \mathtt{IdKernel} \rbrack\!\rbrack &= I_d \\
\lbrack\!\lbrack \mathtt{ZeroKernel} \rbrack\!\rbrack &= 0_{d \times d}
\end{aligned}
$$

结果是一个 $nd \times md$ 的分块矩阵。

**正确性准则**：对于任意两个块矩阵 $F$、$G$，若 $H$ 是按 §4.3 规则展开的 $G \circ F$，则：

$$
\lbrack\!\lbrack H \rbrack\!\rbrack = \lbrack\!\lbrack G \rbrack\!\rbrack \cdot \lbrack\!\lbrack F \rbrack\!\rbrack
$$

此等式保证 eager flattening 是语义保持的。

### 4.5 块矩阵的执行语义

块矩阵 $\mathtt{BlockMat}(n, m, K)$ 作用于输入向量 $\mathbf{x} = (x_0, x_1, \ldots, x_{m-1})$，其中每个 $x_j \in k^d$（一个基础块），产出输出向量 $\mathbf{y} = (y_0, y_1, \ldots, y_{n-1})$：

$$
y_i = \sum_{j=0}^{m-1} K[i][j](x_j) \qquad i = 0, 1, \ldots, n-1
$$

其中 $K[i][j](x_j)$ 表示以 $K[i][j]$ 的参数矩阵左乘 $x_j$。当 $K[i][j] = \mathtt{ZeroKernel}$ 时该项为零，不参与求和。

这个执行语义隐含了全部结构操作：

- 每个非零 $K[i][j]$ 触发一次局部核执行；
- 每个输出 $y_i$ 的求和是一次归约操作（`merge`）；
- 某列出现多个非零块意味着该输入块被多处读取（`copy`/广播）；
- 某列全为零意味着该输入块不被消费（`discard`）。

### 4.6 IR 示例

**示例 1**：$k^6 \to k^4$ 的线性映射（执行模型 §5.4 的例子）

$k^6 = B^{\oplus 3}$，$k^4 = B^{\oplus 2}$，态射为 $2 \times 3$ 块矩阵：

```
f = BlockMat(2, 3, [
  [ Kernel(M_11), Kernel(M_12), Kernel(M_13) ],
  [ Kernel(M_21), Kernel(M_22), Kernel(M_23) ]
])
-- 类型：Blocks(3) → Blocks(2)
```

求值：

$$
\lbrack\!\lbrack f \rbrack\!\rbrack = \begin{pmatrix} M_{11} & M_{12} & M_{13} \\ M_{21} & M_{22} & M_{23} \end{pmatrix} \in k^{4 \times 6}
$$

**示例 2**：恒等态射 $\text{id}_{B^{\oplus 3}}$

```
id3 = BlockMat(3, 3, [
  [ IdKernel,   ZeroKernel, ZeroKernel ],
  [ ZeroKernel, IdKernel,   ZeroKernel ],
  [ ZeroKernel, ZeroKernel, IdKernel   ]
])
```

**示例 3**：块对角态射 $f \oplus g$

```
f_plus_g = BlockMat(3, 3, [
  [ Kernel(M_f), ZeroKernel, ZeroKernel ],
  [ ZeroKernel,  Kernel(M_g1), ZeroKernel ],
  [ ZeroKernel,  ZeroKernel,   Kernel(M_g2) ]
])
-- 表示 f ⊕ g，其中 f : B → B，g : B^{⊕2} → B^{⊕2}
```

**示例 4**：复制 + 分支变换（$\langle f, g \rangle$ 即 $\mathtt{Pair}(f, g)$）

对 $f : B \to B$，$g : B \to B$：

```
pair_fg = BlockMat(2, 1, [
  [ Kernel(M_f) ],
  [ Kernel(M_g) ]
])
-- 类型：Blocks(1) → Blocks(2)
-- 输入块被 f 和 g 分别作用，相当于 copy 后分别执行
```

## 5. 重写与优化规则

### 5.1 设计目标

重写规则操作的对象是 `BlockMat(n, m, K)` 的核引用网格 $K$。目标是识别 $K$ 中的**结构模式**并简化调度图，同时保证 $\lbrack\!\lbrack K \rbrack\!\rbrack$ 不变。

### 5.2 核心重写规则

#### R1. 零块消去

$$
K[i][j] = \mathtt{ZeroKernel} \quad \Longrightarrow \quad \text{从调度图中删除该核实例及其输入/输出连接}
$$

**效果**：该位置不生成核实例，不占用核槽位，不触发归约输入端口。若第 $i$ 行的所有非零块因消去而减少，归约树的规模相应缩小。

**语义保持**：$\mathtt{ZeroKernel}(x_j) = 0$，删除后对 $y_i$ 的求和结果无影响。

#### R2. 恒等块特化

$$
K[i][j] = \mathtt{IdKernel} \quad \Longrightarrow \quad \text{替换为直通连线，不分配核槽位}
$$

**效果**：输入块 $x_j$ 直接作为该核的输出参与第 $i$ 行的归约，不占用核执行资源。在地址映射层实现为输入缓冲到归约端口的直通路径。

**语义保持**：$\mathtt{IdKernel}(x_j) = x_j$。

#### R3. 标量核特化

$$
K[i][j] = \mathtt{Kernel}(c \cdot I_d) \quad \Longrightarrow \quad \text{标记为标量核，使用逐元素乘法器执行}
$$

**效果**：不占用通用乘加单元，只使用逐元素标量乘法器（代价远低于 $d \times d$ 核）。

**语义保持**：$\mathtt{Kernel}(c \cdot I_d)(x_j) = c \cdot x_j$。

#### R4. 参数共享检测

$$
K[i_1][j_1] = \mathtt{Kernel}(M), \quad K[i_2][j_2] = \mathtt{Kernel}(M) \quad (M \text{ 相同})
$$

$$
\Longrightarrow \quad \text{两个核实例共享同一参数存储，各自独立执行}
$$

**效果**：参数存储复用，装载期只需装载一份 $M$。核槽位仍独立分配（因输入不同，计算不可共享）。共享参数使 §6.3 代价公式中的 $N_{\text{unique}}$ 小于 $N_{\text{gen}}$，从而降低 $C_{\text{load}}$。

#### R5. 死列消去

$$
\forall i: \quad K[i][j] = \mathtt{ZeroKernel} \quad \Longrightarrow \quad \text{输入块 } x_j \text{ 不被消费，可跳过读取}
$$

**效果**：该输入块不从输入缓冲加载，节省输入带宽。等价于 `discard` 操作。

**lowering 层自动处理**：附录 A 的 `lower` 函数在构建 `InputReadNode` 时仅为 `liveInputCols`（即至少有一个非零消费者的列）生成读入请求，死列自然不被包含。因此 R5 不需要独立的编译 pass，其语义已由 lowering 逻辑直接保证。

#### R6. 死行消去

$$
\forall j: \quad K[i][j] = \mathtt{ZeroKernel} \quad \Longrightarrow \quad \text{输出块 } y_i = 0\text{，可直接填零或跳过归约}
$$

**效果**：该输出位置不建立归约节点，直接写零或由下游感知稀疏性。

**lowering 层自动处理**：附录 A 的 `lower` 函数中，`outputNodeType []`（即该行无任何非零块）映射为 `ZeroFillNode`，不建立归约节点。因此 R6 与 R5 类似，不需要独立的编译 pass，其语义已由 lowering 逻辑直接保证。

#### R7. 单项行特化

$$
\text{第 } i \text{ 行仅有一个非零块 } K[i][j_0] \quad \Longrightarrow \quad y_i = K[i][j_0](x_{j_0})\text{，无需归约}
$$

**效果**：该输出块不需要归约单元，核输出直接写入输出缓冲。消除归约开销。在调度图中，该行生成 `DirectNode`（直通节点）而非 `ReduceNode`，与附录 A 中 `outputNodeType [_] = DirectNode` 对齐。

#### R8. 置换行/列检测

$$
\text{BlockMat 为置换矩阵（每行每列恰好一个 IdKernel，其余为 ZeroKernel）}
$$

$$
\Longrightarrow \quad \text{整个态射退化为块重排，仅需地址重映射}
$$

**效果**：不执行任何核，不消耗算术资源。在共享缓冲架构下为零成本视图变换。

### 5.3 重写策略

所有规则均为**单向确定性规则**，在编译期对 $K$ 做一次扫描即可全部应用：

1. 遍历 $K$，标记每个核引用为 `Zero`/`Id`/`Scalar`/`General`
2. 检测参数共享（哈希比对）
3. 统计每行/每列的非零块数
4. 应用 R1-R8

**终止性**：所有规则严格递减调度图的节点数或资源占用量，不会循环。

**无双向规则**：第一阶段不存在双向重写问题，因为 IR 中没有可在两个方向变换的组合结构。块矩阵一旦确定，优化完全是确定性的模式识别。

### 5.4 稠密度与结构特征分析

设块矩阵为 $n \times m$，非零块比例为 $\rho = |\{(i,j) : K[i][j] \neq \mathtt{ZeroKernel}\}| / (nm)$。

**第一阶段不做自动展平切换。** 所有验证程序一律走块路线（R1-R8 优化 → 调度图生成 → 执行），以确保在全稠密（$\rho = 1$）到高稀疏（$\rho \ll 1$）的完整区间内收集块路线的代价数据。展平路线仅作为代价对比基线同时计算并输出（见 §6.5），不作为运行时执行通路。

编译器在每次编译时输出以下**结构特征摘要**，作为验证报告的标准组成部分：

| 指标 | 定义 | 用途 |
|------|------|------|
| $\rho$ | 非零块比例 | 稠密度基本度量 |
| $N_{\text{zero}}$ | ZeroKernel 块数 | R1 消去潜力 |
| $N_{\text{id}}$ | IdKernel 块数 | R2 直通潜力 |
| $N_{\text{scalar}}$ | 标量核（$c \cdot I_d$）块数 | R3 特化潜力 |
| $N_{\text{shared}}$ | 共享参数组数 | R4 参数复用潜力 |
| $N_{\text{dead\_col}}$ | 死列数 | R5 输入跳过潜力 |
| $N_{\text{dead\_row}}$ | 死行数 | R6 输出跳过潜力 |
| $N_{\text{single}}$ | 单项行数 | R7 归约消除潜力 |
| 是否置换矩阵 | 布尔 | R8 零成本执行潜力 |

编译器始终同时计算块路线和展平路线的代价估计，输出代价比 $R = C_{\text{block}} / C_{\text{flat}}$。

**后续扩展备忘**：当积累足够的 benchmark 数据后，可引入自动路径切换准则。该准则应至少是 $(\rho, \text{可特化块比例})$ 的二元函数——单一 $\rho$ 会把高稠密但高度结构化（如大量标量核或共享参数）的块矩阵误判为"无结构收益"。

## 6. 代价模型

### 6.1 代价层次

| 层次 | 含义 | 可量化参量 |
|------|------|----------|
| 核执行代价 | 局部核阵列的计算延迟 | 周期数 |
| 结构代价 | 广播、归约、地址映射的运行时开销 | 周期数 |
| 装载代价 | 参数从存储到核寄存器的传输 | 周期数 |

### 6.2 代价参数表

| 操作 | 代价参量 | 单位 | 说明 |
|------|---------|------|------|
| 一般核执行 | $L_{\text{core}}$ | 周期 | 流水线深度，第一阶段目标 $L_{\text{core}} \leq 4$ |
| IdKernel（直通） | 0 | 周期 | 零成本旁路 |
| ZeroKernel（不执行） | 0 | 周期 | 不进入调度 |
| 标量核 | $L_{\text{scale}} \leq L_{\text{core}}$ | 周期 | 逐元素乘法（$d = 2$ 时 $L_{\text{scale}} = 1$） |
| 广播（$n_f$ 路，单端口） | $\lceil \log_2 n_f \rceil$ 或 $n_f$ | 周期 | 广播树 vs 串行复制；$B_{\text{ports}}$ 个端口可并行服务 |
| 归约（$n_r$ 路树形） | $\lceil \log_2 n_r \rceil$ | 周期 | 加法树 |
| 归约（$n_r$ 路流水线） | $n_r$ | 周期 | 单累加器 |
| 参数装载（一般核，每槽位） | $\lceil d^2 / B_w \rceil$ | 周期 | $B_w$：参数总线宽度（标量/周期） |
| 参数装载（标量核，每槽位） | $\lceil 1 / B_w \rceil$ | 周期 | 标量核只需装载 1 个标量（见 §7.2） |
| 输出写回 | $\lceil n \cdot d / W_{\text{out}} \rceil$ | 周期 | $W_{\text{out}}$：输出缓冲写端口带宽（标量/周期） |

### 6.3 单态射总代价估计

对 `BlockMat(n, m, K)`，定义以下符号：

| 符号 | 定义 |
|------|------|
| $N_{\text{nz}}$ | 非零块数（$K[i][j] \neq \mathtt{ZeroKernel}$ 的位置数）|
| $N_{\text{gen}}$ | 需要真实核执行的块数（非 Id 非 Zero）|
| $N_{\text{scale}}$ | 标量核数（R3 识别的 $c \cdot I_d$ 形式核），$N_{\text{scale}} \leq N_{\text{gen}}$ |
| $N_{\text{unique}}$ | 去重后的唯一参数矩阵数（R4 参数共享后的独立参数组数）|
| $r_i$ | 第 $i$ 行的非零块数 |
| $c_j$ | 第 $j$ 列的非零块数（即输入块 $x_j$ 的消费者数）|
| $P$ | 核阵列并行槽位数 |
| $B_{\text{ports}}$ | 广播/路由单元的并行端口数（可同时服务的独立广播树数） |
| $R_{\text{ports}}$ | 可用归约端口数 |

**延迟估计**（关键路径）：

$$
C_{\text{latency}} = C_{\text{load}} + C_{\text{exec}} + C_{\text{bcast}} + C_{\text{reduce}} + C_{\text{wb}}
$$

各项定义如下：

**（1）参数装载 $C_{\text{load}}$**

$$
C_{\text{load}} = \left\lceil \frac{(N_{\text{unique}} - N_{\text{scale\_u}}) \cdot d^2 + N_{\text{scale\_u}} \cdot 1}{P \cdot B_w} \right\rceil
$$

其中 $N_{\text{scale\_u}}$ 是去重后的唯一标量核参数数（每个标量核只需装载 1 个标量，见 §7.2 标量核旁路模型）。当无标量核时 $N_{\text{scale\_u}} = 0$，公式退化为 $\lceil N_{\text{unique}} \cdot d^2 / (P \cdot B_w) \rceil$。当无参数共享时 $N_{\text{unique}} = N_{\text{gen}}$。

**边界情形**：当 $N_{\text{gen}} = 0$（纯结构态射，如纯置换或纯恒等直通）时，$N_{\text{unique}} = 0$，$C_{\text{load}} = 0$，$C_{\text{exec}} = 0$。全部代价退化为结构层操作（广播 + 归约 + 写回），与 R2/R8 的零执行语义一致。

**装载隐藏（双缓冲）**：当核阵列支持双缓冲参数装载时，$C_{\text{load}}$ 可与上一批的 $C_{\text{exec}}$ 重叠。修正后的有效装载代价为：

$$
C_{\text{load}}^{\text{eff}} = \begin{cases}
0 & \text{若 } N_{\text{gen}} = 0 \text{（纯结构态射，无装载）} \\
\max\!\left(0,\; C_{\text{load}} - \max(0,\; N_{\text{batches}} - 1) \cdot L_{\text{batch}}\right) & \text{否则}
\end{cases}
$$

其中 $N_{\text{batches}} = \lceil (N_{\text{gen}} - N_{\text{scale}}) / P \rceil + \lceil N_{\text{scale}} / P \rceil$ 是总批次数（一般核批次 + 标量核批次），$L_{\text{batch}}$ 取各批次中的最小执行延迟（保守取 $L_{\text{scale}}$）。

直觉：当 $N_{\text{gen}} \geq 1$ 时，从第二批开始，每批执行期间可同时装载下一批参数。当批次足够多时，$C_{\text{load}}^{\text{eff}} = 0$。当 $N_{\text{gen}} = 0$ 时无批次、无参数、无装载，结果直接为 0。第一阶段的代价报告应同时输出 $C_{\text{load}}$（无双缓冲）和 $C_{\text{load}}^{\text{eff}}$（有双缓冲）两个值。

**第一阶段建模边界**：双缓冲仅作为解析估算值出现在代价报告中（即 $C_{\text{load}}^{\text{eff}}$）。附录 A 的 `scheduleGraph` 周期推进器中，`ParamLoadNode` 作为单个事务节点建模，不拆分为逐批次的多个装载节点。因此模拟器的 `reportScheduleCycles` 不包含双缓冲的装载隐藏效果——这正是它与 `reportAnalyticTotalEff` 之间可能出现差异的来源之一。若后续需要模拟器精确建模双缓冲，应将 `ParamLoadNode` 拆分为按核批次对齐的多个装载事务，每个事务与上一批核执行并行发射。

**（2）广播 $C_{\text{bcast}}$**

广播代价取决于广播/路由单元的并行端口数 $B_{\text{ports}}$（可同时服务的独立广播树数）。设需要广播的输入列集合为 $J_{\text{bcast}} = \{j : c_j > 1\}$。

$$
C_{\text{bcast}} = \left\lceil \frac{|J_{\text{bcast}}|}{B_{\text{ports}}} \right\rceil \cdot \max_{j \in J_{\text{bcast}}} C_{\text{bcast},j}
$$

直觉：$|J_{\text{bcast}}|$ 列需要广播，每批最多 $B_{\text{ports}}$ 列并行广播，每批延迟取该批中最慢一列。保守估计取全局最慢列作为每批延迟的上界。当 $B_{\text{ports}} \geq |J_{\text{bcast}}|$ 时（端口充足），所有列并行广播，$C_{\text{bcast}} = \max_j C_{\text{bcast},j}$。当 $B_{\text{ports}} = 1$ 时（单端口），所有列串行广播，$C_{\text{bcast}} = |J_{\text{bcast}}| \cdot \max_j C_{\text{bcast},j}$。

其中每列的广播延迟取决于该输入块的消费者数 $c_j$：

$$
C_{\text{bcast},j} = \begin{cases}
0 & \text{若 } c_j \leq 1 \text{（无需广播）} \\
\lceil \log_2 c_j \rceil & \text{若广播树可用} \\
c_j & \text{若串行复制}
\end{cases}
$$

**（3）核执行 $C_{\text{exec}}$**

核执行按类型分为一般核批次和标量核批次，分别以不同延迟通过核阵列：

$$
C_{\text{exec}} = \left\lceil \frac{N_{\text{gen}} - N_{\text{scale}}}{P} \right\rceil \cdot L_{\text{core}} + \left\lceil \frac{N_{\text{scale}}}{P} \right\rceil \cdot L_{\text{scale}}
$$

第一项为一般核（包含所有非标量、非 Id、非 Zero 的核）的执行延迟，第二项为标量核的执行延迟。编译器将标量核和一般核分到不同批次调度，以充分利用 $L_{\text{scale}} < L_{\text{core}}$ 的延迟优势。当 $N_{\text{scale}} = 0$ 时退化为 $\lceil N_{\text{gen}} / P \rceil \cdot L_{\text{core}}$；当所有核均为标量核时退化为 $\lceil N_{\text{gen}} / P \rceil \cdot L_{\text{scale}}$。

**（4）归约 $C_{\text{reduce}}$**

归约代价同时建模行内归约延迟和行间端口争用。设需要归约的行集合为 $I_{\text{reduce}} = \{i : r_i > 1\}$（$r_i = 1$ 的行无需归约，见 R7）。

$$
C_{\text{reduce}} = \left\lceil \frac{|I_{\text{reduce}}|}{R_{\text{ports}}} \right\rceil \cdot \max_{i \in I_{\text{reduce}}} C_{\text{reduce},i}
$$

直觉：$|I_{\text{reduce}}|$ 行需要归约，每批最多 $R_{\text{ports}}$ 行并行归约，每批延迟取该批中最慢一行。保守估计取全局最慢行作为每批延迟上界。当 $R_{\text{ports}} \geq |I_{\text{reduce}}|$ 时（端口充足），所有行并行归约，$C_{\text{reduce}} = \max_i C_{\text{reduce},i}$。当 $R_{\text{ports}} = 1$ 时（单端口），所有行串行归约。

其中每行的归约延迟由该行的非零块数 $r_i$ 和归约单元内部结构决定：

$$
C_{\text{reduce},i} = \begin{cases}
0 & \text{若 } r_i \leq 1 \text{（无需归约，R7 单项行）} \\
\lceil \log_2 r_i \rceil & \text{若树形归约（每端口内加法器充足）} \\
r_i & \text{若流水线累加（单累加器）}
\end{cases}
$$

树形 vs 流水线的选择由每个归约端口内部的加法器数量决定（设计参数），与端口数 $R_{\text{ports}}$ 正交。

**（5）写回 $C_{\text{wb}}$**

$$
C_{\text{wb}} = \left\lceil \frac{n \cdot d}{W_{\text{out}}} \right\rceil
$$

其中 $W_{\text{out}}$ 是输出缓冲写端口的标量/周期带宽。当 $W_{\text{out}} \geq n \cdot d$ 时（写端口足够宽），$C_{\text{wb}} = 1$ 周期，可被流水线隐藏。

**第一阶段写回粒度**：第一阶段将写回建模为单个批量事务（即整个调度图仅生成一个 `WriteBackNode`，依赖所有输出行节点），不做逐行流水线写回。这是有意的保守建模：实际中行级归约完成后即可立即写回，拆分为逐行 `WriteBackNode` 可进一步降低延迟。后续若需更精确的写回建模，可将每个输出行单独生成一个 `WriteBackNode`，延迟模型由 $\lceil n \cdot d / W_{\text{out}} \rceil$ 退化为 $\lceil d / W_{\text{out}} \rceil$。

**关于流水线重叠的说明**：上述公式将各阶段串行相加，给出保守的延迟上界。在实际调度中，每批广播完成后该批相关核即可发射（广播与核执行部分重叠），归约也可在核输出流到达时流水线式启动。第一阶段的模拟器应输出串行上界和实际调度周期两个值，以量化流水线重叠的收益。

**解析公式与模拟器调度的关系定位**：`reportScheduleCycles`（附录 A `scheduleGraph` 的实际周期级模拟输出）是行为基准值，`reportAnalyticTotalEff`（本节封闭公式）是保守上界。当 `reportScheduleCycles` < `reportAnalyticTotalEff` 时，差值即流水线重叠带来的收益。代价报告中两者均输出，以供对照和解释。

### 6.4 算术强度

单次核调用的算术强度（执行模型 §7.5）：

$$
I(d) = \frac{d^2}{(2d + d^2) \cdot W} = \frac{1}{(2/d + 1) \cdot W}
$$

取 $W = 2$（FP16）：

| $d$ | MAC 数 | 读写字节 | $I(d)$ (MAC/Byte) |
|-----|--------|---------|-------------------|
| 2 | 4 | 16 | 0.25 |
| 4 | 16 | 48 | 0.33 |
| 8 | 64 | 160 | 0.40 |
| 16 | 256 | 576 | 0.44 |

**$d = 2$ 确认为通信受限**。第一阶段的目标不是达到高算术强度，而是验证方法论。验证报告应包含此曲线。

### 6.5 代价对比框架

对同一态射 $f : B^{\oplus m} \to B^{\oplus n}$，给出两条路线的代价：

- **块路线**：按上述 §6.3 计算
- **展平路线**：将整个块矩阵视为单一 $nd \times md$ 矩阵-向量乘法，使用与块路线**相同的微架构参数**（$P$、$L_{\text{core}}$、$B_w$、$B_{\text{ports}}$、$R_{\text{ports}}$、$W_{\text{out}}$）计算代价。展平意味着所有 $nm$ 个 $d \times d$ 子块均视为一般核（无零块消去、无恒等块特化、无标量核优化、无参数共享），归约固定为每行 $m$ 项。封闭公式如下：

$$
C_{\text{flat}} = C_{\text{load,flat}} + C_{\text{exec,flat}} + C_{\text{bcast,flat}} + C_{\text{reduce,flat}} + C_{\text{wb,flat}}
$$

其中：

$$
\begin{aligned}
C_{\text{load,flat}} &= \left\lceil \frac{nm \cdot d^2}{P \cdot B_w} \right\rceil \quad \text{（全部 } nm \text{ 个核的参数装载，无参数共享）} \\
C_{\text{exec,flat}} &= \left\lceil \frac{nm}{P} \right\rceil \cdot L_{\text{core}} \quad \text{（全部 } nm \text{ 个核均按一般核执行，无标量核特化）} \\
C_{\text{bcast,flat}} &= \left\lceil \frac{m}{B_{\text{ports}}} \right\rceil \cdot C_{\text{bcast},j}(n) \quad \text{（每列固定 } n \text{ 个消费者，共 } m \text{ 列需广播，按端口数分批）} \\
C_{\text{reduce,flat}} &= \left\lceil \frac{n}{R_{\text{ports}}} \right\rceil \cdot C_{\text{reduce},i}(m) \quad \text{（每行归约 } m \text{ 项，共 } n \text{ 行，按 } R_{\text{ports}} \text{ 分批）} \\
C_{\text{wb,flat}} &= \left\lceil \frac{n \cdot d}{W_{\text{out}}} \right\rceil \quad \text{（写回代价，与块路线相同）}
\end{aligned}
$$

此公式与块路线 §6.3 完全同量纲（周期数）、同参数集、同代价项结构，使代价比 $R$ 可计算、可解释。

代价比 $R = C_{\text{block}} / C_{\text{flat}}$ 作为验收报告的核心指标。$R < 1$ 表示块路线更优（结构消去带来净收益），$R > 1$ 表示块路线更贵（结构开销超过消去收益），$R = 1$ 表示两条路线等价。

**展平基线的语义边界（重要）**：展平路线是“同微架构、同块粒度、不利用结构信息”的基线，用于隔离度量“结构利用 vs 不利用”的增量收益。它**不是**传统密集矩阵引擎（如 GPU tensor core 或 systolic array）的最优基线——后者通常在更大块宽、更高算术强度的工作点上运行，与本系统的 $d = 2$ 微架构不在同一比较平面上。因此，$R < 1$ 证明的是“结构消去在本微架构上有净收益”，而不是“块架构比传统密集引擎更快”。后一类结论需要等到更大块宽（$d \geq 8$）且与真实密集基线对照后才能得出。

**块宽升级的工程影响**（对齐执行模型 §7.5）：当块宽从 $d = 2$ 升级到 $d \geq 4$ 时，算术密度从 $O(d)$ 量级提升，代价模型中 $L_{\text{core}}$、$C_{\text{load}}$ 等参数需相应更新，但结构层与调度层的接口协议不变。

### 7.2 局部核阵列

| 规格 | 第一阶段定义 |
|------|-------------|
| 核槽位数 | $P$（设计参数，建议 4–8） |
| 核规格 | $d \times d$ 矩阵-向量乘（$d = 2$ 时为 2×2） |
| 参数存储 | 每槽位 $d^2$ 个标量的寄存器文件 |
| 执行延迟 | $L_{\text{core}}$ 周期（目标 $\leq 4$） |
| 数值格式 | FP32（验证）/ FP16（对比）|
| 输入端口 | $d$ 个标量读端口，接共享缓冲 |
| 输出端口 | $d$ 个标量写端口，接归约单元或输出缓冲 |
| 核后非线性 | 可旁路的固定功能单元（ReLU / 截断） |
| 标量核旁路 | 每槽位配备可旁路的逐元素标量乘法器（见下文） |

**核执行模型**：每个核槽位在一个时钟批次中执行一次 $d \times d$ 矩阵-向量乘。$P$ 个槽位可并行执行 $P$ 个独立核。当 BlockMat 中非零核数 $N_{\text{gen}} > P$ 时，分 $\lceil N_{\text{gen}} / P \rceil$ 批执行。

**标量核旁路模型**：当编译器标记某核为标量核（R3，参数为 $c \cdot I_d$）时，该槽位旁路通用乘加 datapath，改用逐元素标量乘法器执行 $v \mapsto c \cdot v$。具体而言：

- **参数装载**：只需装载 1 个标量 $c$（而非 $d^2$ 个），降低装载带宽占用；
- **执行延迟**：$L_{\text{scale}} \leq L_{\text{core}}$，$d = 2$ 时 $L_{\text{scale}} = 1$ 周期（两个并行乘法器）；
- **硬件开销**：每槽位增加 $d$ 个标量乘法器和一个旁路 MUX，在 $d = 2$ 时面积可忽略。

该旁路使 R3 从纯编译器标签提升为硬件层面的真实特化。

### 7.3 结构模块

每个模块的职责与接口：

**视图/地址生成单元**

- 功能：根据调度图中的块偏移信息，生成核输入端口的读地址和输出端口的写地址
- 接口：接收调度表 → 输出（块索引, 缓冲偏移）对
- 吸收操作：`proj`、`swap`、部分 `inj` 的地址偏移

**广播/路由单元**

- 功能：当某输入块被多个核读取时（块矩阵某列有多个非零块），将该块数据广播到多个核输入端口
- 接口：（源块地址, 目标核槽位列表）→ 数据分发
- 并行度：$B_{\text{ports}}$ 个独立广播端口可同时服务不同输入列的广播；超出端口数的列按批次串行处理
- 实现：广播树或多端口读

**归约单元**

- 功能：对每个输出行执行 $y_i = \sum_j K[i][j](x_j)$ 的求和；同时承载结构操作 `merge` 的 lower 语义（`merge(n)` → `reduce_sum`，对齐执行模型 §6.2）
- 接口：（核输出流, 输出块索引）→ $d$ 维归约结果
- 并行度：$R_{\text{ports}}$ 个独立归约端口可同时服务不同输出行的归约；超出端口数的行按批次串行处理
- 模式（每端口内部）：
  - 树形归约：$\lceil \log_2 r_i \rceil$ 级加法树，$r_i - 1$ 个加法器
  - 流水线累加：单累加器逐块累加
- 选择：编译器根据每行 $r_i$ 和端口内部加法器数量决定

**零填充与写回单元**

- 功能：处理 `inj` 的补零语义，输出块汇集与写回到输出缓冲
- 特殊情况：死行（R6）直接填零

**缓冲与生命周期管理**

- 功能：临时块的分配、复用与释放
- 核心约束：输入块在所有消费它的核执行完毕后释放；输出块在归约完成后写回

### 7.4 共享缓冲区

| 分区 | 容量要求 | 用途 |
|------|---------|------|
| 输入块缓冲 | $m \cdot d$ 个标量 | 存放 $m$ 个输入块 |
| 输出块缓冲 | $n \cdot d$ 个标量 | 存放 $n$ 个输出块 |
| 临时块缓冲 | $\geq P \cdot d$ 个标量 | 核输出暂存，等待归约 |
| 参数存储 | $(N_{\text{unique}} - N_{\text{scale\_u}}) \cdot d^2 + N_{\text{scale\_u}}$ 个标量（或分批装载） | 去重后的参数；标量核每个只占 1 标量 |

第一阶段总缓冲容量目标：足以容纳验证例（$k^8 \to k^6$，即 $m = 4, n = 3, d = 2$）的全部数据，约 $4 \times 2 + 3 \times 2 + P \times 2 + 12 \times 4 = 14 + 2P + 48$ 个标量。当 $P = 8$ 时约 78 个标量，远低于任何实际 SRAM 容量。

### 7.5 一次执行的生命周期

1. **编译期**：块矩阵 IR → R1-R8 优化 → 生成调度图（核实例列表 + 归约配置 + 广播配置 + 地址映射）
2. **装载期**：将非零核参数写入核槽位寄存器文件；地址映射和归约配置写入结构模块控制寄存器。装载延迟按 §6.3 $C_{\text{load}}$ 公式计算（已计入参数共享和标量核装载减少）。支持双缓冲时，装载可与上一批执行重叠
3. **执行期**：
   - 输入数据按块写入输入缓冲
   - 广播/路由单元按调度图将输入块分发到核输入端口
   - 核阵列并行执行，一般核和标量核分别按 $\lceil (N_{\text{gen}} - N_{\text{scale}}) / P \rceil$ 和 $\lceil N_{\text{scale}} / P \rceil$ 批调度
   - 每批核输出进入归约单元
   - 归约完成后输出块写入输出缓冲
4. **回收期**：释放临时缓冲，核阵列就绪

### 7.6 新核沉淀准则（对齐执行模型 §6.6）

统一执行模型并不排斥新增硬件核，但新增核必须以统一母体为前提。本节以第一阶段的**标量核旁路**（§7.2，R3）为范例，说明新核沉淀的判定流程与接口约束。

#### 7.6.1 四项定性准则

一个被识别出的结构，只有同时满足以下条件时，才值得从"编译期特化模式"进一步沉淀为"后端专用核"：

1. **结构稳定**：该结构在多个任务或多个编译样例中反复出现，而不是一次性模式。
2. **收益明确**：在面积、延迟、带宽或能效上，相对通用局部核或通用结构路径有可量化优势。
3. **接口继承**：新核仍然服从统一块级输入输出协议（$d$ 维块输入/$d$ 维块输出）、参数装载语义和调度契约（统一进入 `scheduleGraph`）。
4. **组合兼容**：新核仍然能作为统一调度图中的普通节点，与结构原语、缓冲和其他核族共同组合。

#### 7.6.2 标量核旁路作为沉淀范例

| 准则 | 标量核旁路的满足方式 |
|------|------------------|
| 结构稳定 | $c \cdot I_d$ 形式的核在对角矩阵、残差连接、缩放层等场景中高频出现 |
| 收益明确 | 参数装载从 $d^2$ 降至 1 个标量，执行延迟从 $L_{\text{core}}$ 降至 $L_{\text{scale}}$；代价报告可量化差异 |
| 接口继承 | 标量核仍经由 `CoreNode (ScalarKernel c)` 进入调度图，输入输出仍为 $d$ 维块，参数通过统一的 `paramHandle` 装载 |
| 组合兼容 | 标量核与一般核在同一调度图中可混合调度，归约单元不感知上游核类型 |

#### 7.6.3 定量参照框架

一个候选结构至少应在不少于 $N$ 个代表性编译样例中稳定出现，且相对通用核路径的代价收益比不低于阈值 $\rho$（如面积×延迟不高于通用路径的 $\rho$ 倍），方可进入沉淀流程。$N$ 和 $\rho$ 的具体取值应由第一阶段实验数据校准。

第一阶段的代价报告（§8.4 V1–V8）应包含以下扩核相关指标：

| 指标 | 定义 | 用途 |
|------|------|------|
| 标量核命中率 | $N_{\text{scale}} / N_{\text{gen}}$ | 评估标量核旁路的结构稳定性 |
| 标量核装载节省 | $(N_{\text{scale\_u}} \cdot (d^2 - 1)) / (N_{\text{unique}} \cdot d^2)$ | 量化参数装载收益 |
| 标量核延迟节省 | $(L_{\text{core}} - L_{\text{scale}}) \cdot N_{\text{scale}} / P$ | 量化执行延迟收益 |

## 8. 闭环验证路径

### 8.1 验证架构

```
结构化程序构造（§3.5 构造函数）
     │
     │  proj / inj / pair / copair / copy / merge / bimap / compose
     ▼
┌────────────────┐
│ BlockMat IR    │  §4 定义的 AST（唯一 IR）
└───────┬────────┘
        │  R1-R8 优化
        ▼
┌────────────────┐     ┌────────────────────┐
│ 优化后 BlockMat│────→│ 矩阵求值器 (oracle) │
└───────┬────────┘     │ eval() → k^{nd×md} │
        │              │     (FP32)          │
        │  调度图生成    └─────────┬──────────┘
        ▼                        │
┌────────────────┐               │
│ 调度图执行      │               │
│ (模拟器/硬件)   │               │
└───────┬────────┘               │
        │                        │
        ▼                        ▼
  执行结果 (nd 维向量)     oracle 结果 (nd 维向量)
        │                        │
        └───────── 对照 ─────────┘
                   ‖结果 - oracle‖ < ε (FP32)
```

### 8.2 三阶段验证

**阶段一：求值器 + 构造函数闭环**

- 实现 BlockMat IR 数据结构和 §3.5 构造函数（proj, inj, pair, copair, copy, merge, bimap, compose）
- 实现 §4.4 求值函数 `eval(BlockMat) → Matrix`
- 验证构造函数产出的 BlockMat 满足结构等式 S1-S7（§3.5.3）
- 对 eager flattening 验证 $\lbrack\!\lbrack G \circ F \rbrack\!\rbrack = \lbrack\!\lbrack G \rbrack\!\rbrack \cdot \lbrack\!\lbrack F \rbrack\!\rbrack$
- 底域：$k = \mathbb{R}$（FP32）；等式验证使用相对误差 $< 10^{-6}$

**验收标准**：（a）示例 1（§4.6）的 $2 \times 3$ 块矩阵在求值器中跑通；（b）用构造函数构造 `pair(kernelApp(M_f), kernelApp(M_g))` 并验证其 `eval()` 结果与手工计算一致。

**阶段二：优化器闭环**

- 实现 §5.2 的 R1-R8 优化 pass
- 验证"优化前后 `eval()` 结果不变"
- 对至少一个示例，展示零块消去（R1）和恒等块特化（R2）的效果

**验收标准**：在示例 1 中手动设置 $M_{13} = 0$、$M_{11} = I_d$，验证：
- R1 消去 $K[0][2]$
- R2 将 $K[0][0]$ 特化为直通
- 优化前后 `eval()` 数值一致

**阶段三：执行器闭环**

- 实现周期级模拟器（基于 §7 的模块划分）
- 编译后的调度图在模拟器中执行，产出数值结果 + 代价报告
- 数值结果与求值器 oracle 对照

**验收标准**：$k^8 \to k^6$（$4 \times 3$ 块矩阵）端到端跑通，结果与 oracle 一致，代价报告包含延迟和资源使用两个维度。

### 8.3 六条验收条件

1. **端到端正确性**：至少一个 $4 \times 3$ 块矩阵（$k^8 \to k^6$）从 IR 编译到执行后端，产出与直接矩阵乘一致的数值结果。

2. **结构优化可观测**：编译器能识别并消去至少一种结构冗余（零块消去或恒等块特化），且消去前后数值结果一致。

3. **分离执行可观测**：调度图中结构模块与局部核由不同模块/阶段独立执行——即广播、归约、核执行在模拟器中有独立的周期轨迹。

4. **代价可对比**：同一态射给出块路线 vs 展平路线的代价比 $R = C_{\text{block}} / C_{\text{flat}}$，可量化，差异来源应可追溯到具体的结构优化规则编号。

5. **结构等式可验证**：构造函数产出的 BlockMat 满足 S1-S7 等式（§3.5.3）——例如 `compose(proj(i,n), inj(i,n))` 的 `eval()` 结果等于单位矩阵，`compose(merge(n), copy(n))` 的 `eval()` 结果等于 $n \cdot I_d$。此条件证明"结构操作的块矩阵编码是数学正确的"。

6. **扩核契约可保持**（对齐执行模型 §8 条件 5）：若引入一个结构化专用核作为实验性扩展（第一阶段以标量核旁路 §7.2 为实例），该核仍必须复用同一块级表示（`CoreNode ExecKernelRef`）、输入输出接口（$d$ 维块输入/输出）和调度语义（统一进入 `scheduleGraph` 调度），而不是旁路式特例接口。验证方式：在标量核旁路启用/禁用两种配置下，同一块矩阵的调度图均可正常生成和执行，且数值结果一致。

### 8.4 代表性验证程序

| 编号 | 程序 | 块矩阵形状 | 特征 | 重点验证 |
|------|------|-----------|------|---------|
| V1 | 全稠密基准 | $2 \times 3$ | $\rho = 1.0$ | 端到端正确性、归约调度、块路线在最不利条件下的代价上界 |
| V2 | 含零块 | $2 \times 3$，$K[0][2] = 0$ | $\rho = 5/6$ | R1 零块消去 |
| V3 | 含恒等块 | $2 \times 3$，$K[0][0] = \text{Id}$ | — | R2 恒等块特化 |
| V4 | 块对角 | $3 \times 3$ 对角 | $\rho = 1/3$ | R7 单项行特化、结构利用度 |
| V5 | 置换 | $3 \times 3$ 置换矩阵 | $\rho = 1/3$ | R8 置换检测、零成本执行 |
| V6 | 大规模 | $4 \times 3$（$k^8 \to k^6$） | 混合 | 完整闭环 + 代价对比 |
| V7 | 双积分解加法 | $2 \times 2$ | 由 `morphAdd` 组装的局部态射和 | 双积诱导加法、S7 关键等式、结构化组装的端到端闭环 |
| V8 | 128 维多结构混合 | $64 \times 64$（$k^{128} \to k^{128}$） | 单一案例内混合 Zero/Id/Scalar/共享参数/死行死列/单项行/高扇出广播与多项归约 | 上百维规模下联合覆盖 R1-R7 的主要结构收益场景，并与展平基线做代价对比 |

**V8 构造规则**：$64 \times 64$ 块矩阵按以下六个区段构造，确保单一案例内覆盖全部主要结构模式。预定义 6 个核矩阵：$M_{\times 2} = 2I_d$、$M_{\text{neg}} = -I_d$、$M_{\text{half}} = 0.5 I_d$（标量核）；$M_{\text{shear}} = \begin{pmatrix}1&1\\0&1\end{pmatrix}$、$M_{\text{swap}} = \begin{pmatrix}0&1\\1&0\end{pmatrix}$、$M_{\text{mix}} = \begin{pmatrix}1&-1\\2&0\end{pmatrix}$（一般核）。

- **区段 A**（行 $0 \leq i < 16$）：$K[i][i] = \mathtt{IdKernel}$，其余为零。每行恰好单项 Id，覆盖 R2 恒等块特化 + R7 单项行特化。
- **区段 B**（行 $16 \leq i < 32$）：设 $b = i - 16$，$K[i][b] = M_{\times 2}$，$K[i][16+b] = M_{\text{neg}}$，其余为零。每行双项标量核归约，覆盖 R3 标量核检测 + R4 参数共享（$M_{\times 2}$ 和 $M_{\text{neg}}$ 分别被 16 行共享）。
- **区段 C**（行 $32 \leq i < 48$）：设 $b = i - 32$，按如下规则散列填充三个位置：$K[i][(2b) \bmod 56] = M_{\text{shear}}$，$K[i][(2b+5) \bmod 56] = M_{\text{swap}}$，$K[i][(2b+9) \bmod 56] = \begin{cases} M_{\text{half}} & b \text{ 偶数} \\ \mathtt{IdKernel} & b \text{ 奇数} \end{cases}$。每行三项混合核（一般核 + 标量核 + Id），制造中等扇出与共享参数复用，覆盖 R4 参数共享 + 混合结构行归约。
- **区段 D**（行 $48 \leq i < 56$）：每行固定四个位置 $K[i][2] = M_{\text{shear}}$，$K[i][18] = M_{\text{mix}}$，$K[i][34] = M_{\times 2}$，$K[i][50] = M_{\text{swap}}$。8 行同时写入列 2/18/34/50，形成高扇出广播热点（每列有 8+$\alpha$ 个消费者）；每行四项归约，覆盖高扇出广播与多项归约。
- **区段 E**（行 $56 \leq i < 60$）：$K[i][i] = M_{\text{mix}}$，其余为零。单项一般核行，覆盖 R7 单项行特化，并保持列 56–59 为活列。
- **区段 F**（行 $60 \leq i < 64$）：整行保持全零，作为死行（覆盖 R6 死行消去）。列 60–63 因无任何行消费而成为死列（覆盖 R5 死列消去）。

**实测结构统计**（与 `benchmarks.rs` 测试断言对齐）：$N_{\text{nz}} = 132$，$N_{\text{unique}} = 6$，$N_{\text{scale\_u}} = 3$，$N_{\text{scale}} = 48$，dead\_cols $= 22$，dead\_rows $= 4$，single\_rows $= 20$，reduce\_rows $= 40$，bcast\_cols $= 36$，peak\_input\_buffer $= 42$。

参考实现可在 `benchmarks.rs` 的 `v8_hundred_dim_banded()` 中查阅。

## 附录 A：IR 的代数数据类型定义（Haskell 风格伪代码）

```haskell
-- ===== 参数 =====
d :: Nat
d = 2  -- 第一阶段块宽

-- ===== 类型 =====
data Ty = Blocks Nat       -- B^{⊕n}，n ≥ 1
  deriving (Eq, Show)

dim :: Ty -> Nat
dim (Blocks n) = n * d

-- ===== 核引用 =====
data KernelRef
  = Kernel (Matrix d d)    -- 一般 d×d 参数矩阵
  | IdKernel               -- 恒等核 I_d
  | ZeroKernel             -- 零核 0_{d×d}
  deriving (Eq, Show)

-- ===== 态射项 =====
data Morph = BlockMat
  { rows :: Nat                        -- n（输出块数）
  , cols :: Nat                        -- m（输入块数）
  , grid :: Vector (Vector KernelRef)  -- n × m 网格
  } deriving (Eq, Show)

-- ===== 结构验证与类型推断 =====
data ValidationError
  = NonPositiveRows
  | NonPositiveCols
  | RowCountMismatch
  | ColCountMismatch
  | EmptyBimap
  | PairDomainMismatch
  | CopairCodomainMismatch
  deriving (Eq, Show)

validateMorph :: Morph -> Either ValidationError Morph
validateMorph morph@(BlockMat n m grid)
  | n < 1                  = Left NonPositiveRows
  | m < 1                  = Left NonPositiveCols
  | length grid /= n       = Left RowCountMismatch
  | any ((/= m) . length) grid = Left ColCountMismatch
  | otherwise              = Right morph

typeOf :: Morph -> Either ValidationError (Ty, Ty)
typeOf morph = do
  BlockMat n m _ <- validateMorph morph
  pure (Blocks m, Blocks n)

-- ===== 核引用级别运算 =====

-- 核组合（矩阵乘法）
kernelCompose :: KernelRef -> KernelRef -> KernelRef
kernelCompose ZeroKernel _          = ZeroKernel
kernelCompose _ ZeroKernel          = ZeroKernel
kernelCompose IdKernel k            = k
kernelCompose k IdKernel            = k
kernelCompose (Kernel m1) (Kernel m2) = Kernel (m1 `matMul` m2)

-- 核求和（矩阵加法）
kernelAdd :: KernelRef -> KernelRef -> KernelRef
kernelAdd ZeroKernel k          = k
kernelAdd k ZeroKernel          = k
kernelAdd (Kernel m1) (Kernel m2) = Kernel (m1 `matAdd` m2)
kernelAdd IdKernel (Kernel m)   = Kernel (identityMatrix d `matAdd` m)
kernelAdd (Kernel m) IdKernel   = Kernel (m `matAdd` identityMatrix d)
kernelAdd IdKernel IdKernel     = Kernel (scalarMatrix 2 d)  -- 2·I_d

-- ===== Eager Flattening =====
compose :: Morph -> Morph -> Morph  -- compose g f = g ∘ f
compose (BlockMat p n kg) (BlockMat n' m kf)
  | n /= n'   = error "dimension mismatch"
  | otherwise  = BlockMat p m result
  where
    result = [[ foldl1 kernelAdd
                  [ kernelCompose (kg !! i !! j) (kf !! j !! k)
                  | j <- [0..n-1] ]
              | k <- [0..m-1] ]
             | i <- [0..p-1] ]

-- ===== 求值语义 =====
evalKernel :: KernelRef -> Matrix d d
evalKernel (Kernel m) = m
evalKernel IdKernel   = identityMatrix d
evalKernel ZeroKernel = zeroMatrix d d

eval :: Morph -> Matrix (n*d) (m*d)
eval (BlockMat n m grid) = blockMatrix
  [[ evalKernel (grid !! i !! j) | j <- [0..m-1] ]
                                 | i <- [0..n-1] ]

-- ===== 调度图生成（Lowering）=====
type NodeId = Int

data ExecKernelRef = GeneralKernel (Matrix d d)
                   | ScalarKernel Scalar
                   | IdExec
                   | ZeroExec

data NodeType = InputReadNode             -- 输入块读入输入缓冲
              | ParamLoadNode             -- 参数块装载/共享参数准备
              | ViewAddrNode              -- 视图/地址生成（编译期静态偏移映射，对应 proj/部分 pair/swap/discard）
              | BroadcastNode             -- 一列输入被多处消费时的扇出
              | RouteSelectNode           -- 路由选择（copair 的分支选择，见 §3.5 copair lowering）
                                          -- 【第一阶段预留】：当前 copair 退化为横向拼接，由 ReduceNode
                                          -- 统一承载每行归约。RouteSelectNode 不在第一阶段 lower 函数中
                                          -- 实例化，但保留类型定义，以便后续支持显式 copair 多路选择。
              | CoreNode ExecKernelRef    -- 通用/标量核执行
              | EltwiseNode               -- 逐元素非线性（对齐执行模型 §6.3 core_eltwise）
                                          -- 【第一阶段预留】：第一阶段不在 lower 中生成此节点，
                                          -- 但保留类型定义以对齐执行模型原语枚举。
                                          -- 启用后：输入 1 个 d 维块，输出 1 个 d 维块，
                                          -- 延迟 L_eltwise，可旁路。
              | DirectNode                -- 直通连线（IdKernel 或单输入行）
              | ReduceNode                -- 多输入归约（含 merge → reduce_sum）
              | ZeroFillNode              -- 全零行补零
              | DiscardReleaseNode        -- 丢弃释放（discard 的缓冲释放）
              | WriteBackNode             -- 输出缓冲写回
              -- === 运行时控制原语（对齐执行模型 §6.4）===
              -- 【第一阶段预留】：静态调度模型下这些原语多数可在编译期确定，
              -- 第一阶段 lower 不生成以下节点。保留类型定义以便后续启用
              -- 缓冲区压力、端口冲突和生命周期的显式事件建模。
              | BarrierSyncNode           -- 同步屏障（一组事件完成后解除下游阻塞）
              | SlotReserveNode           -- 槽位预分配（临时缓冲分配）
              | SlotReleaseNode           -- 槽位释放（消费者全部完成后回收）

data SchedNode = SchedNode
  { nodeId        :: NodeId
  , nodeType      :: NodeType
  , arity         :: Int            -- 扇入/扇出度，用于延迟模型
  , transferWidth :: Int            -- 本节点搬运的标量数
  , inputSlots    :: [BufferAddr]     -- 输入块地址
  , outputSlots   :: [BufferAddr]     -- 输出块地址
  , paramHandle   :: Maybe ParamAddr  -- 参数地址（仅 CoreNode）
  , dependencies  :: [NodeId]         -- 上游依赖（避免节点对象递归嵌套）
  }

data SchedGraph = SchedGraph
  { nodes       :: Map NodeId SchedNode
  , outputNodes :: [NodeId]             -- 每个输出行对应的“行结果节点”
  , permuteMap  :: Maybe [Int]          -- R8：纯置换时记录 output[i] <- input[permuteMap[i]]
  }

-- R8: 置换矩阵检测
isPermutationMorph :: Morph -> Bool
isPermutationMorph (BlockMat n m grid) =
  n == m
  && all (== 1) rowCounts
  && all (== 1) colCounts
  && all isIdOrZero (concat grid)
  where
    rowCounts =
      [ length [ () | k <- row, k /= ZeroKernel ]
      | row <- grid
      ]
    colCounts =
      [ length [ () | i <- [0..n-1], grid !! i !! j /= ZeroKernel ]
      | j <- [0..m-1]
      ]
    isIdOrZero IdKernel   = True
    isIdOrZero ZeroKernel = True
    isIdOrZero _          = False

permutationMap :: Morph -> [Int]
permutationMap (BlockMat n _ grid) =
  [ head [ j | j <- [0..n-1], grid !! i !! j == IdKernel ]
  | i <- [0..n-1]
  ]

-- lower : Morph -> Either ValidationError SchedGraph
lower :: Morph -> Either ValidationError SchedGraph
lower morph@(BlockMat n m grid) = do
  _ <- validateMorph morph
  if isPermutationMorph morph
    then pure SchedGraph
      { nodes = Map.empty
      , outputNodes = []
      , permuteMap = Just (permutationMap morph)
      }
    else
      let nonZeroInCol j = [ i | i <- [0..n-1], grid !! i !! j /= ZeroKernel ]
          nonZeroInRow i = [ j | j <- [0..m-1], grid !! i !! j /= ZeroKernel ]
          liveInputCols = [ j | j <- [0..m-1], not (null (nonZeroInCol j)) ]
          uniqueParams = collectUniqueParams grid

      -- 第一阶段的 lowering 顺序：
      -- 1. 若存在活跃输入列，则建立一个 InputReadNode；
      -- 2. 若存在可共享参数，则建立一个 ParamLoadNode；
      -- 3. 为需要广播的输入列建立 BroadcastNode；
      -- 4. 为每个非零块建立 CoreNode 或 DirectNode；
      -- 5. 按输出行建立 ReduceNode / DirectNode / ZeroFillNode；
      --    copair lowering 说明（对齐执行模型 §6.2）：
      --      第一阶段 copair 统一走“横向拼接 + 行归约”路径：
      --      copair(f, g) 产出 BlockMat(n, p+q, ...)，每行多个非零块由 ReduceNode 统一归约。
      --      RouteSelectNode 在第一阶段不实例化，保留类型定义以便后续支持显式 copair 多路选择。
      --      未来若需支持显式 copair 多路选择：
      --        - 若 copair 的所有分支指向同一目标块 → lower 为 RouteSelectNode（纯路由选择）
      --        - 若分支指向不同目标块需汇合 → lower 为 RouteSelectNode + ReduceNode（选择 + 归约）
      -- 6. 非纯置换时建立一个 WriteBackNode；
      -- 7. outputNodes 仅记录第 5 步得到的“行结果节点”。

      inputReadNode =
        if null liveInputCols then [] else [mkInputReadNode 0 liveInputCols]

      paramLoadNode =
        if null uniqueParams then [] else [mkParamLoadNode (length inputReadNode) uniqueParams]

      buildBroadcastNodes =
        [ (j, BroadcastNode)
        | j <- [0..m-1]
        , length (nonZeroInCol j) > 1
        ]

      buildKernelNodes =
        [ ((i, j), kernelNodeType (execKernelRef (grid !! i !! j)))
        | i <- [0..n-1]
        , j <- [0..m-1]
        , grid !! i !! j /= ZeroKernel
        ]

      buildOutputNodes =
        [ (i, outputNodeType (nonZeroInRow i))
        | i <- [0..n-1]
        ]

      -- 具体实现中可用 State NodeId 分配器替代 zip [0..]。
      inputReadBase = length inputReadNode
      paramLoadBase = inputReadBase + length paramLoadNode

      broadcastIds = Map.fromList
        [ (j, nodeId')
        | (nodeId', (j, _)) <- zip [paramLoadBase..] buildBroadcastNodes
        ]

      kernelBase = paramLoadBase + length buildBroadcastNodes
      kernelIds = Map.fromList
        [ ((i, j), nodeId')
        | (nodeId', ((i, j), _)) <- zip [kernelBase..] buildKernelNodes
        ]

      outputBase = kernelBase + length buildKernelNodes
      outputIds = Map.fromList
        [ (i, nodeId')
        | (nodeId', (i, _)) <- zip [outputBase..] buildOutputNodes
        ]

      writeBackBase = outputBase + length buildOutputNodes

      broadcastNodes =
        [ mkBroadcastNode nodeId' inputReadId j
        | (nodeId', (j, _)) <- zip [paramLoadBase..] buildBroadcastNodes
        ]

      kernelNodes =
        [ mkKernelNode nodeId' i j ref (Map.lookup j broadcastIds) (kernelDeps i j)
        | (nodeId', ((i, j), _)) <- zip [kernelBase..] buildKernelNodes
        , let ref = grid !! i !! j
        ]

      outputNodes' =
        [ mkOutputNode nodeId' i (outputNodeType (nonZeroInRow i)) (rowDeps i)
        | (nodeId', (i, _)) <- zip [outputBase..] buildOutputNodes
        ]

      writeBackNodes =
        if null outputNodes'
          then []
          else [mkWriteBackNode writeBackBase [ outputIds Map.! i | i <- [0..n-1] ]]

      inputReadId = listToMaybe [ nodeId n | n <- inputReadNode ]
      paramLoadId = listToMaybe [ nodeId n | n <- paramLoadNode ]

      kernelDeps i j =
        let fromBroadcast = maybeToList (Map.lookup j broadcastIds)
            fromInputRead = if null fromBroadcast then maybeToList inputReadId else []
            fromParamLoad =
              if needsParam (execKernelRef (grid !! i !! j))
                then maybeToList paramLoadId
                else []
        in fromBroadcast ++ fromInputRead ++ fromParamLoad

      rowDeps i =
        [ kernelIds Map.! (i, j)
        | j <- nonZeroInRow i
        ]

      in pure SchedGraph
        { nodes = Map.fromList
            [ (nodeId node, node)
            | node <- inputReadNode ++ paramLoadNode ++ broadcastNodes ++ kernelNodes ++ outputNodes' ++ writeBackNodes
            ]
        , outputNodes = [ outputIds Map.! i | i <- [0..n-1] ]
        , permuteMap = Nothing
        }

execKernelRef :: KernelRef -> ExecKernelRef
execKernelRef = undefined

needsParam :: ExecKernelRef -> Bool
needsParam IdExec            = False
needsParam ZeroExec          = False
needsParam _                 = True

collectUniqueParams :: [[KernelRef]] -> [ParamAddr]
collectUniqueParams = undefined

kernelNodeType :: ExecKernelRef -> NodeType
kernelNodeType IdExec       = DirectNode
kernelNodeType ZeroExec     = error "ZeroKernel should be removed before lowering"
kernelNodeType execRef      = CoreNode execRef

outputNodeType :: [Int] -> NodeType
outputNodeType []    = ZeroFillNode
outputNodeType [_]   = DirectNode   -- 对应 R7：单项行无需归约
outputNodeType _     = ReduceNode

mkInputReadNode :: NodeId -> [Int] -> SchedNode
mkInputReadNode nodeId' liveCols = SchedNode
  { nodeId = nodeId'
  , nodeType = InputReadNode
  , arity = length liveCols
  , transferWidth = length liveCols * d
  , inputSlots = map inputAddr liveCols
  , outputSlots = map inputAddr liveCols
  , paramHandle = Nothing
  , dependencies = []
  }

mkParamLoadNode :: NodeId -> [ParamAddr] -> SchedNode
mkParamLoadNode nodeId' params = SchedNode
  { nodeId = nodeId'
  , nodeType = ParamLoadNode
  , arity = length params
  , transferWidth = sum (map paramScalars params)
  , inputSlots = []
  , outputSlots = []
  , paramHandle = Nothing
  , dependencies = []
  }

mkBroadcastNode :: NodeId -> Maybe NodeId -> Int -> SchedNode
mkBroadcastNode nodeId' inputReadId inputBlock = SchedNode
  { nodeId = nodeId'
  , nodeType = BroadcastNode
  , arity = 1
  , transferWidth = d
  , inputSlots = [inputAddr inputBlock]
  , outputSlots = [tempAddr nodeId']
  , paramHandle = Nothing
  , dependencies = maybeToList inputReadId
  }

mkKernelNode :: NodeId -> Int -> Int -> KernelRef -> Maybe NodeId -> [NodeId] -> SchedNode
mkKernelNode nodeId' outRow inCol kernelRef maybeBcastId deps = SchedNode
  { nodeId = nodeId'
  , nodeType = kernelNodeType (execKernelRef kernelRef)
  , arity = 1
  , transferWidth = 0
  , inputSlots = case maybeBcastId of
      Just bcastId -> [tempAddr bcastId]
      Nothing      -> [inputAddr inCol]
  , outputSlots = [tempAddr nodeId']
  , paramHandle = kernelParam (execKernelRef kernelRef)
  , dependencies = deps
  }

mkOutputNode :: NodeId -> Int -> NodeType -> [NodeId] -> SchedNode
mkOutputNode nodeId' outRow nodeType' deps = SchedNode
  { nodeId = nodeId'
  , nodeType = nodeType'
  , arity = length deps
  , transferWidth = 0
  , inputSlots = map tempAddr deps
  , outputSlots = [outputAddr outRow]
  , paramHandle = Nothing
  , dependencies = deps
  }

mkWriteBackNode :: NodeId -> [NodeId] -> SchedNode
mkWriteBackNode nodeId' deps = SchedNode
  { nodeId = nodeId'
  , nodeType = WriteBackNode
  , arity = length deps
  , transferWidth = length deps * d
  , inputSlots = map tempAddr deps
  , outputSlots = map outputAddr [0..length deps - 1]
  , paramHandle = Nothing
  , dependencies = deps
  }

kernelParam :: ExecKernelRef -> Maybe ParamAddr
kernelParam (GeneralKernel _) = Just freshParamAddr
kernelParam (ScalarKernel _)  = Just freshParamAddr
kernelParam IdExec            = Nothing
kernelParam ZeroExec          = Nothing

paramScalars :: ParamAddr -> Int
paramScalars = undefined

-- inputAddr / outputAddr / tempAddr / freshParamAddr 为后端相关 helper：
-- 它们把块索引映射到共享缓冲区地址或参数存储句柄。
-- 若 permuteMap = Just p，则运行时无需发射任何算术/结构节点，
-- 只需按 p 做输入块到输出块的地址重映射即可；这就是 R8 的零成本执行路径。
-- outputNodes 始终对应“行结果节点”而非最终 WriteBackNode，
-- 这样模拟器与代价报告都可以把“行内归约完成”与“统一写回完成”分开统计。

-- ===== 最小调度器框架（Scheduling）=====
data LatencyModel = Tree
                  | Pipeline
                  | Fixed Int

data SchedParams = SchedParams
  { coreSlots      :: Int          -- P：核阵列槽位数
  , bcastPorts     :: Int          -- B_ports：广播端口数
  , reducePorts    :: Int          -- R_ports：归约端口数
  , coreLatency    :: Int          -- L_core
  , scaleLatency   :: Int          -- L_scale
  , directLatency  :: Int          -- IdKernel 直通延迟（第一阶段可取 0 或 1）
  , bwParams       :: Int          -- B_w：参数装载带宽（标量/周期）
  , bwWriteback    :: Int          -- W_out：输入/输出搬运带宽（标量/周期）
  , bcastModel     :: LatencyModel -- 广播延迟模型
  , reduceModel    :: LatencyModel -- 归约延迟模型
  }

data Batch = Batch
  { batchCycle :: Int
  , batchNodes :: [NodeId]
  } deriving (Eq, Show)

data TimedNode = TimedNode
  { timedId      :: NodeId
  , startCycle   :: Int
  , finishCycle  :: Int
  } deriving (Eq, Show)

data Schedule = Schedule
  { topoOrder     :: [NodeId]
  , issuedBatches :: [Batch]
  , timedNodes   :: [TimedNode]
  , totalCycles  :: Int
  } deriving (Eq, Show)

-- topoSort : 使用 Kahn 算法按依赖关系得到稳定拓扑序。
-- 第一阶段建议按 NodeId 升序打破并列，保证调度结果可复现。
topoSort :: SchedGraph -> [NodeId]
topoSort graph = kahn initialInDegree initialReady []
  where
    ns = Map.elems (nodes graph)
    initialInDegree = Map.fromList
      [ (nodeId n, length (dependencies n)) | n <- ns ]
    initialReady = sort
      [ nodeId n | n <- ns, null (dependencies n) ]

    usersOf target =
      [ nodeId n
      | n <- ns
      , target `elem` dependencies n
      ]

    kahn indegree [] acc = reverse acc
    kahn indegree (n:ready) acc =
      let (indegree', newlyReady) = foldl (relax n) (indegree, []) (usersOf n)
      in kahn indegree' (sort (ready ++ newlyReady)) (n:acc)

    relax finished (indegree, readyAcc) user =
      let deg = indegree Map.! user - 1
          indegree' = Map.insert user deg indegree
      in if deg == 0
           then (indegree', user : readyAcc)
           else (indegree', readyAcc)

-- resourceClass : 把节点映射到调度资源类。
resourceClass :: NodeType -> String
resourceClass InputReadNode    = "input"
resourceClass ParamLoadNode    = "load"
resourceClass ViewAddrNode     = "none"     -- 视图生成，编译期吸收
resourceClass BroadcastNode    = "bcast"
resourceClass RouteSelectNode  = "bcast"    -- 复用广播端口资源（预留）
resourceClass (CoreNode _)     = "core"
resourceClass EltwiseNode      = "core"     -- 复用核槽位资源（预留）
resourceClass DirectNode       = "none"
resourceClass ReduceNode       = "reduce"
resourceClass ZeroFillNode     = "reduce"
resourceClass DiscardReleaseNode = "none"   -- 丢弃释放
resourceClass WriteBackNode    = "writeback"
resourceClass BarrierSyncNode  = "none"     -- 零周期控制判定（预留）
resourceClass SlotReserveNode  = "none"     -- 零周期控制更新（预留）
resourceClass SlotReleaseNode  = "none"     -- 零周期控制更新（预留）

nodeLatency :: SchedParams -> SchedNode -> Int
nodeLatency params node = case nodeType node of
  InputReadNode  -> ceilDiv (transferWidth node) (bwWriteback params)
  ParamLoadNode  -> ceilDiv (transferWidth node) (coreSlots params * bwParams params)
  ViewAddrNode   -> 0                                                    -- 视图生成，编译期吸收
  BroadcastNode  -> latency (bcastModel params) (max 1 (arity node))
  RouteSelectNode -> latency (bcastModel params) (max 1 (arity node))    -- 预留，复用广播延迟模型
  CoreNode (ScalarKernel _) -> scaleLatency params
  CoreNode _     -> coreLatency params
  EltwiseNode    -> scaleLatency params                                  -- 预留；启用后可独立配置 L_eltwise
  DirectNode     -> directLatency params
  ReduceNode     -> latency (reduceModel params) (arity node)
  ZeroFillNode   -> 1
  DiscardReleaseNode -> 0                                                -- 丢弃释放
  WriteBackNode  -> ceilDiv (transferWidth node) (bwWriteback params)
  BarrierSyncNode -> 0                                                   -- 预留，零周期控制判定
  SlotReserveNode -> 0                                                   -- 预留，零周期控制更新
  SlotReleaseNode -> 0                                                   -- 预留，零周期控制更新

latency :: LatencyModel -> Int -> Int
latency Tree k = if k <= 1 then 0 else ceilLog2 k
latency Pipeline k = k
latency (Fixed c) _ = c

-- readyNodes : 当前周期所有依赖均已完成、且尚未发射的节点。
readyNodes :: SchedGraph -> Map NodeId Int -> Set NodeId -> Int -> [NodeId]
readyNodes graph finishTimes issued now =
  sort
    [ nodeId node
    | node <- Map.elems (nodes graph)
    , nodeId node `Set.notMember` issued
    , all (\dep -> Map.findWithDefault maxBound dep finishTimes <= now) (dependencies node)
    ]

-- issueBatch : 在单个周期内按资源上限选择可发射节点。
-- 第一阶段采用贪心策略：按 ready 队列顺序扫描，资源未满就发射。
-- 其中 InputRead / ParamLoad / WriteBack 先按“每周期至多 1 个事务”建模；
-- DirectNode 不单独占用功能部件，但仍受依赖与时序约束。
issueBatch :: SchedGraph -> SchedParams -> [NodeId] -> [NodeId]
issueBatch graph params ready = go ready 0 0 0 0 0 0 []
  where
    go [] _ _ _ _ _ _ acc = reverse acc
    go (nid:rest) usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc =
      let node = nodes graph Map.! nid
      in case resourceClass (nodeType node) of
           "input" ->
             if usedInput < 1
               then go rest (usedInput + 1) usedLoad usedCore usedBcast usedReduce usedWriteBack (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc
           "load" ->
             if usedLoad < 1
               then go rest usedInput (usedLoad + 1) usedCore usedBcast usedReduce usedWriteBack (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc
           "core" ->
             if usedCore < coreSlots params
               then go rest usedInput usedLoad (usedCore + 1) usedBcast usedReduce usedWriteBack (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc
           "bcast" ->
             if usedBcast < bcastPorts params
               then go rest usedInput usedLoad usedCore (usedBcast + 1) usedReduce usedWriteBack (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc
           "writeback" ->
             if usedWriteBack < 1
               then go rest usedInput usedLoad usedCore usedBcast usedReduce (usedWriteBack + 1) (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc
           "none" ->
             go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack (nid:acc)
           _ ->
             if usedReduce < reducePorts params
               then go rest usedInput usedLoad usedCore usedBcast (usedReduce + 1) usedWriteBack (nid:acc)
               else go rest usedInput usedLoad usedCore usedBcast usedReduce usedWriteBack acc

-- scheduleGraph : 最小周期推进器。
-- 语义：每个周期收集 ready 节点，按资源上限发射一批，并记录完成周期。
scheduleGraph :: SchedGraph -> SchedParams -> Schedule
scheduleGraph graph params = loop 0 Set.empty Map.empty [] []
  where
    allNodeIds = Map.keysSet (nodes graph)

    loop now issued finishTimes batches timed
      | issued == allNodeIds = Schedule
          { topoOrder = topoSort graph
          , issuedBatches = reverse batches
          , timedNodes = reverse timed
          , totalCycles = now
          }
      | otherwise =
          let ready = readyNodes graph finishTimes issued now
              batch = issueBatch graph params ready
          in if null batch
               then loop (now + 1) issued finishTimes batches timed
               else
                 let newTimed =
                       [ TimedNode
                           { timedId = nid
                           , startCycle = now
                           , finishCycle = now + nodeLatency params (nodes graph Map.! nid)
                           }
                       | nid <- batch
                       ]
                     finishTimes' = foldl
                       (\acc t -> Map.insert (timedId t) (finishCycle t) acc)
                       finishTimes
                       newTimed
                     issued' = foldl (flip Set.insert) issued batch
                 in loop (now + 1)
                         issued'
                         finishTimes'
                         (Batch now batch : batches)
                         (reverse newTimed ++ timed)

-- 说明：
-- 1. topoSort 给出稳定的依赖顺序，便于测试和调试；
-- 2. scheduleGraph 给出第一阶段所需的最小周期级原型，不追求最优调度；
-- 3. 若要逼近 §6 的代价模型，可在 issueBatch 中加入“标量核优先”或“同列广播聚合”等启发式规则；
-- 4. 模拟器可直接消费 timedNodes，对每个节点在 [startCycle, finishCycle) 区间内标记资源占用；

-- ===== 结构构造函数（§3.5）=====
-- 每个函数直接产出 Morph（BlockMat），不引入独立的中间表示

kernelApp :: Matrix d d -> Morph
kernelApp m = BlockMat 1 1 [[Kernel m]]

proj :: Int -> Int -> Morph
proj i n = BlockMat 1 n
  [[ if j == i then IdKernel else ZeroKernel | j <- [0..n-1] ]]

inj :: Int -> Int -> Morph
inj i n = BlockMat n 1
  [[ if j == i then IdKernel else ZeroKernel ] | j <- [0..n-1] ]

copy :: Int -> Morph
copy n = BlockMat n 1 [[ IdKernel ] | _ <- [0..n-1] ]

merge :: Int -> Morph
merge n = BlockMat 1 n [[ IdKernel | _ <- [0..n-1] ]]

identity :: Int -> Morph
identity n = BlockMat n n
  [[ if i == j then IdKernel else ZeroKernel
   | j <- [0..n-1] ] | i <- [0..n-1] ]

pair :: Morph -> Morph -> Morph
pair f g = either (error . show) id (verticalConcat f g)

copair :: Morph -> Morph -> Morph
copair f g = either (error . show) id (horizontalConcat f g)

bimap :: [Morph] -> Morph
bimap fs = either (error . show) id (blockDiagonal fs)

-- compose 已定义于 Eager Flattening 部分（见上方）

-- verticalConcat  : 纵向拼接两个块矩阵的核网格（Pair 语义）
verticalConcat :: Morph -> Morph -> Either ValidationError Morph
verticalConcat f@(BlockMat p m gf) g@(BlockMat q m' gg) = do
  _ <- validateMorph f
  _ <- validateMorph g
  if m /= m'
    then Left PairDomainMismatch
    else pure (BlockMat (p + q) m (gf ++ gg))

-- horizontalConcat: 横向拼接两个块矩阵的核网格（Copair 语义）
horizontalConcat :: Morph -> Morph -> Either ValidationError Morph
horizontalConcat f@(BlockMat n p gf) g@(BlockMat n' q gg) = do
  _ <- validateMorph f
  _ <- validateMorph g
  if n /= n'
    then Left CopairCodomainMismatch
    else pure (BlockMat n (p + q)
          [ gf !! i ++ gg !! i | i <- [0..n-1] ])

-- blockDiagonal   : 块对角排列多个块矩阵（BiMap 语义）
blockDiagonal :: [Morph] -> Either ValidationError Morph
blockDiagonal [] = Left EmptyBimap
blockDiagonal fs = do
  traversed <- traverse validateMorph fs
  let rs = map rows traversed
      cs = map cols traversed
      totalRows = sum rs
      totalCols = sum cs
      zeroRow width = [ ZeroKernel | _ <- [1..width] ]
      buildRowBlocks i morph =
        let leftPad  = sum (take i cs)
            rightPad = totalCols - leftPad - cols morph
        in [ zeroRow leftPad ++ row ++ zeroRow rightPad
           | row <- grid morph ]
      result = concat [ buildRowBlocks i morph | (i, morph) <- zip [0..] traversed ]
  pure (BlockMat totalRows totalCols result)
```
