# 微架构接口规格（v0.1）

> 版本：v0.1
>
> 状态：草案

## 0. 文档定位

本文档是第一阶段从“可表示、可编译、可调度、可验证”的软件原型，向“可写 RTL / 可做模块级软仿真”的过渡规格。

它回答的问题不是：

- BlockMat 的数学语义是什么；
- R1-R8 的优化规则是什么；
- 周期级模拟器为什么要这样组织；

这些问题已经分别由以下文档回答：

- [2-执行模型说明-0v2.md](2-执行模型说明-0v2.md)
- [3-技术规格说明-0v2.md](3-技术规格说明-0v2.md)
- [4-模拟器技术选型说明-0v1.md](4-模拟器技术选型说明-0v1.md)

本文档只回答一个更靠近实现的问题：

> **如果要把当前第一阶段的 BlockMat 执行模型继续推进到 RTL 或模块级软实现，那么微架构应如何划分模块、定义接口、编码命令、约定仲裁和时序语义？**

因此，这是一份 **RTL 前置接口规格**，不是数学 spec，也不是物理实现规格。

## 1. 适用范围与边界

本文档适用于第一阶段对象域：

- 基础块固定为 $B = k^d$ ，第一阶段取 $d = 2$
- 对象为有限直和 $B^{\oplus n}$
- 态射为块矩阵 `BlockMat(n, m, K)`
- 编译结果为 schedule-driven 的节点执行表

本文档当前覆盖：

- 模块划分
- 模块间接口
- 地址/句柄逻辑编码
- 调度表与命令记录格式
- 广播/归约/写回的仲裁规则
- 第一阶段逐周期握手约定

本文档当前不覆盖：

- SRAM 宏实例与物理 banking
- 时钟树、复位树、DFT、低功耗门控
- AXI/TileLink 等系统级总线封装
- 面积/功耗/Fmax 收敛
- 第二阶段及以后更大块宽、张量积侧原语

## 2. 设计目标

接口规格必须满足以下约束：

1. 与 [3-技术规格说明-0v2.md](3-技术规格说明-0v2.md) 第 7 章的模块职责一致。
2. 与当前 `csimulator` 的调度图节点类型一一对应，避免语义断层。
3. 可以先支撑模块级软实现，再平滑迁移到 RTL。
4. 第一阶段优先追求确定性和可验证性，不追求最激进吞吐。

由此导出两条总原则：

1. **静态调度优先**：第一阶段由编译器离线生成调度表，运行时不做复杂动态重排。
2. **全局停顿优先于局部乱序**：如果模块反压导致某条命令无法在预定周期发射，第一阶段允许控制器插入 bubble 并整体顺延，而不是立即引入复杂的动态重调度。

## 3. 顶层模块划分

第一阶段推荐的顶层模块如下：

| 模块 | 职责 | 对应当前抽象节点/阶段 |
| ---- | ---- | ---------------------- |
| `DispatchCtrl` | 读取调度表、按周期发射命令、收集完成事件 | `Schedule` 回放 |
| `BufferManager` | 统一管理输入/临时/输出块缓冲的地址空间与分配状态 | 共享缓冲概念 |
| `InputReader` | 将输入块读入执行域 | `InputRead` |
| `ParamLoader` | 按 `param_handle` 装载一般核/标量核参数 | `ParamLoad` |
| `AddrGen` | 把逻辑块地址映射为缓冲访问地址 | `input_slots` / `output_slots` |
| `BroadcastRouter` | 处理多消费者输入块的扇出 | `Broadcast` |
| `KernelArray` | 执行一般核、标量核和直通路径 | `Core` / `Direct` |
| `ReduceUnit` | 对同一输出行的多项贡献做归约 | `Reduce` |
| `ZeroFillUnit` | 生成死行的零块输出 | `ZeroFill` |
| `WriteBackUnit` | 将最终输出写回输出缓冲 | `WriteBack` |
| `LifecycleCtrl` | 追踪临时块和输入块的最后消费者，执行释放 | 当前 `remaining_consumers` 语义 |

当前 `csimulator` 中的节点类型定义见 [csimulator/src/lower.rs](../csimulator/src/lower.rs#L83)。本规格要求未来模块级软实现与 RTL 实现都以该节点集合作为语义锚点。

## 4. 统一接口约定

### 4.1 时钟与复位

所有模块使用统一时钟域：

- `clk`
- `rst_n`

第一阶段默认单时钟域，不引入异步跨域。

### 4.2 命令握手

所有执行模块统一使用 `valid/ready` 风格的命令接口：

- `cmd_valid`
- `cmd_ready`
- `cmd_payload`

当 `cmd_valid && cmd_ready` 同时为高时，命令在当前周期被该模块接受。

### 4.3 完成事件握手

所有执行模块统一使用事件回传接口：

- `evt_valid`
- `evt_payload`

第一阶段可不要求 `evt_ready`，默认 `DispatchCtrl` 始终能吸收完成事件；若后续 trace 或 debug 通路需要节流，再补 `evt_ready`。

### 4.4 数据粒度

基础块宽固定为 $d = 2$ ，因此一个块载荷为两个标量：

```text
BlockPayload = { lane0, lane1 }
```

第一阶段逻辑上定义：

- 正确性路径可用 FP32 表示
- 性能路径可单独映射到 FP16 宽度估算

RTL 实现时可把 `BlockPayload` 参数化为：

```text
scalar_width = 32   // 验证配置
block_width  = 2 * scalar_width
```

### 4.5 逻辑地址空间

为避免第一阶段过早绑定物理 SRAM 组织，地址首先定义为逻辑地址：

```text
BufferAddr = { space, index }
```

其中：

- `space = 0`: input buffer
- `space = 1`: output buffer
- `space = 2`: temp buffer
- `space = 3`: 保留

`index` 的位宽由实现规模决定。第一阶段建议至少支持 256 个块索引。

> **与当前模拟器代码的对齐说明**：当前 `lower.rs` 使用扁平整数编码逻辑地址：`input_addr(j) = j`（0–9999）、`output_addr(i) = 10000 + i`、`temp_addr(nid) = 20000 + nid`。本规格将其规范化为 `{space, index}` 结构化编码，语义一一对应。模块级软实现时两种编码可互转：`space = addr / 10000`，`index = addr % 10000`。

### 4.6 参数句柄

参数句柄定义为：

```text
ParamHandle = { kind, index }
```

其中：

- `kind = 0`: 一般核参数，存 4 个标量
- `kind = 1`: 标量核参数，存 1 个标量
- `kind = 2..3`: 保留

`index` 是参数存储中的逻辑槽位号。

> **与当前模拟器代码的对齐说明**：当前 `lower.rs` 使用扁平 `ParamAddr = usize` 计数器（R4 去重后递增分配）。`kind` 信息隐含于 `ExecKernelRef`（`General` → `kind=0`，`Scalar` → `kind=1`）。本规格将其显式化以便 RTL 区分参数存储宽度。

## 5. 调度表与命令编码

### 5.1 总体思路

第一阶段不在硬件中重新求 ready 集，也不在硬件中重做依赖解析。编译器输出的 schedule 直接被编码为运行时调度表，由 `DispatchCtrl` 按周期回放。

因此，运行时控制面至少包含两张表：

1. `BatchTable`：描述每个周期应发射的命令范围
2. `NodeTable`：描述每条命令的具体内容

### 5.2 BatchTable

每条 batch 记录至少包含：

| 字段 | 含义 |
| ---- | ---- |
| `cycle` | 目标发射周期 |
| `first_node` | 本批第一条命令在 `NodeTable` 中的索引 |
| `node_count` | 本批命令数量 |

### 5.3 NodeTable

每条 node 记录建议至少包含：

| 字段 | 含义 |
| ---- | ---- |
| `node_id` | 调试与事件对照用的逻辑节点编号 |
| `opcode` | `INPUT_READ / PARAM_LOAD / BCAST / CORE / DIRECT / REDUCE / ZERO_FILL / WRITEBACK`，预留 `VIEW_ADDR / ROUTE_SELECT / ELTWISE / BARRIER_SYNC / SLOT_RESERVE / SLOT_RELEASE / DISCARD_RELEASE` 对齐技术规格 §7.0 |
| `src0_addr` | 主输入块地址 |
| `src1_addr` | 次输入块地址或保留 |
| `dst_addr` | 输出块地址 |
| `param_handle` | 参数句柄；非参数类命令可置空 |
| `arity` | 广播/归约扇入扇出度 |
| `aux_list_base` | 辅助输入列表基址（见下文）；非多输入命令可置空 |
| `flags` | 保留位，用于区分标量核、直通等 |

对于多输入命令（归约、写回等），第一阶段引入 `AuxInputTable`：

| 字段 | 含义 |
| ---- | ---- |
| `addr` | 一个输入块地址 |

`NodeTable` 的 `aux_list_base + arity` 定义了该命令在 `AuxInputTable` 中的连续区间。单输入和双输入命令可不使用此表（直接用 `src0_addr` / `src1_addr`）。

对于多输入归约，第一阶段采用 `AuxInputTable` 间接引用方案（即上述 `aux_list_base + arity`）。

作为可选的 RTL 优化，实现时也可限制每条归约命令最多引用固定数量输入，多余部分拆成树状多条 Reduce 命令。但接口规格层面统一以 `AuxInputTable` 为标准格式。

## 6. 模块接口定义

### 6.1 DispatchCtrl

职责：

- 维护全局 `cycle`
- 读取 `BatchTable` / `NodeTable`
- 把命令分发到对应模块
- 接收完成事件并写 trace / status

输入：

- `clk`, `rst_n`
- `start`
- `batch_mem_rdata`
- `node_mem_rdata`
- `aux_mem_rdata`（AuxInputTable 存储读数据）
- `consumer_count_mem_rdata`（ConsumerCountTable 存储读数据；初始化时加载到 LifecycleCtrl）
- 各模块 `evt_valid/evt_payload`

输出：

- 各模块 `cmd_valid/cmd_payload`（每个执行模块类型一个独立的 cmd 端口，总共 8 组）
- `done`
- `error`

> **并行发射**：DispatchCtrl 对每个执行模块类型（InputReader / ParamLoader / BroadcastRouter / KernelArray / ReduceUnit / ZeroFillUnit / WriteBackUnit / LifecycleCtrl）拥有独立的 cmd 端口。一个 batch 内的多条命令在同一周期并行发射到各自的目标模块。

> **同周期多批次**：当前调度器在同一 `cycle` 可产生多个 Batch（由 0 延迟级联触发）。因此 `BatchTable` 中允许多个条目拥有相同 `cycle` 值。DispatchCtrl 对同周期的多个 batch 按 `first_node` 顺序依次处理并发射。

> **error 語义**：`error` 信号在且仅在以下情况置高：（1）某条命令的 `opcode` 不被任何已实现模块识别；（2）在 `BatchTable` 遍历完毕后仍有未完成的 batch（超时）。第一阶段 `error` 为诊断信号，不触发复杂恢复逻辑。

第一阶段控制策略：

- 如果某一批中某条命令目标模块 `cmd_ready = 0`，则 `DispatchCtrl` 停顿当前批的发射，整机 `cycle` 继续前进或冻结必须在实现中二选一。
- 推荐采用 **冻结语义**：当某批未完全发射时，`cycle` 不前进，保持“调度周期 = 机器周期”的可解释性。

### 6.2 BufferManager

职责：

- 提供输入/临时/输出三类逻辑地址空间
- 管理 temp buffer 的 alloc/free
- 管理 output buffer 的写入占用状态

输入：

- 分配请求：`alloc_valid`, `alloc_kind`
- 释放请求：`free_valid`, `free_addr`
- 读写访问请求（来自各执行模块）

输出：

- `alloc_ready`, `alloc_addr`
- 读数据返回
- 写完成确认

第一阶段建议：

- `input buffer` 与 `output buffer` 采用静态索引
- 仅 `temp buffer` 需要动态分配表

> **多端口并发模型**：执行期间多个模块可能在同一周期同时访问 BufferManager（BroadcastRouter 读 input、KernelArray 读/写 temp、ReduceUnit 读多个 temp、WriteBackUnit 写 output、LifecycleCtrl 释放 temp/input）。**第一阶段 BufferManager 假设为理想化无限带宽模型**——任意数量的并发读写请求均可在同一周期完成，不产生结构冒险。物理实现阶段再根据 SRAM banking 和端口数引入仲裁与冲突检测。

### 6.3 InputReader

职责：

- 按输入块列表把 host / preload 数据写入 `input buffer`

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `input_list_base` | 活跃输入列索引列表在 `AuxInputTable` 中的起始位置 |
| `block_count` | 本次读取的输入块数（即活跃列数） |

> **注意**：R1 消零后，活跃输入列可能不连续（例如 $m = 4$ 但列 1 为全零列被消去）。因此使用索引列表而非 `base + count` 连续区间。若第一阶段所有输入列均活跃，列表退化为 `[0, 1, ..., m-1]`。

完成事件：

- `node_id`
- `loaded_count`

### 6.4 ParamLoader

职责：

- 将去重后的参数批量写入共享参数存储（Shared Parameter Store）
- 各 CoreNode 在执行时通过 `param_handle` 索引读取参数

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `param_count` | 本次装载的去重参数组数 |
| `total_scalars` | 总标量数（决定装载延迟 $\lceil \mathrm{total\_scalars} / (P \cdot B_w) \rceil$ ） |

> **与代码对齐**：当前 `lower.rs` 中 `ParamLoad` 节点用 `arity = unique_param_keys.len()` 和 `transfer_scalars = sum(各参数标量数)` 描述装载量。`param_count` 对应 `arity`，`total_scalars` 对应 `transfer_scalars`。参数的 R4 去重在编译期完成，各 CoreNode 的 `param_handle` 索引指向共享参数存储中的逻辑槽位。

完成事件：

- `node_id`
- `param_count`

### 6.5 AddrGen

职责：

- 把逻辑 `BufferAddr` 转换为 `BufferManager` 所需的具体地址与 bank 选择

第一阶段建议：

- `AddrGen` 可以先做成组合逻辑 helper，而不是独立时序模块
- 但接口规格中保留其模块地位，便于后续加入 bank conflict 检测

### 6.6 BroadcastRouter

职责：

- 从一个输入块产生多个目标消费者的数据副本

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `src_addr` | 源块地址（input buffer 中的输入列） |
| `dst_addr` | 广播结果写入的临时块地址 |
| `fanout` | 下游消费者数（用于延迟模型计算） |

> **广播语义说明**：当前模拟器中，每个 `BroadcastNode` 将一个输入块数据复制到一个临时缓冲地址（`temp_addr(nid)`），下游多个 Core/Direct 节点通过依赖关系各自读取该临时地址。因此，广播不是"写入多个目标"，而是"写一份到 temp、多个消费者通过依赖追踪各自读取"。`LifecycleCtrl` 负责在所有消费者完成后释放该临时块。

完成事件：

- `node_id`
- `fanout`

仲裁规则：

- 第一阶段有 `B_ports` 个独立广播端口
- 同一周期最多发射 `B_ports` 条广播命令
- 同一源块的多条广播命令不自动合并；合并由编译器承担

### 6.7 KernelArray

职责：

- 执行一般核、标量核和直通数据路径

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `slot_id` | 目标核槽位 |
| `src_addr` | 输入块地址 |
| `dst_addr` | 输出临时块地址 |
| `param_handle` | 参数句柄；直通时为空 |
| `exec_kind` | `GENERAL / SCALAR / DIRECT` |

> **slot_id 分配**：当前调度器按贪心策略动态分配核槽位，分配结果不持久化到 `Schedule`。在生成 `NodeTable` 时，编译器应为每条 `CORE` 命令显式记录 `slot_id`，以便 RTL 对参数寄存器文件做精确读写。`DIRECT` 类命令不占用核槽位（`resource_class = "none"`），`slot_id` 置空。

完成事件：

- `node_id`
- `dst_addr`

时序约定：

- `GENERAL` 延迟为 $L_{core}$
- `SCALAR` 延迟为 $L_{scale}$
- `DIRECT` 延迟可取 0 或 1；若 RTL 不支持 0 周期，统一取 1 周期并在报告中单独口径化

### 6.8 ReduceUnit

职责：

- 对同一输出行的多个临时块求和

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `aux_list_base` | 输入块地址列表在 `AuxInputTable` 中的起始位置 |
| `arity` | 归约输入数 |
| `dst_addr` | 结果地址 |
| `reduce_mode` | `TREE / PIPELINE` |

> `AuxInputTable[aux_list_base .. aux_list_base + arity]` 中依次存放各贡献核输出的临时块地址，与当前代码 `ReduceNode.input_slots: Vec<BufferAddr>` 语义对应。

完成事件：

- `node_id`
- `dst_addr`

仲裁规则：

- 第一阶段有 `R_ports` 个归约端口
- 同一周期最多发射 `R_ports` 条归约命令
- 端口内模式固定后，延迟由 `arity` 决定

### 6.9 ZeroFillUnit

职责：

- 为死行生成零块输出

> **资源共享**：当前模拟器中 `ZeroFill` 与 `Reduce` 共享 `reduce` 资源类别（`resource_class(ZeroFill) = "reduce"`），即 ZeroFill 也占用 `R_ports` 归约端口。硬件实现中 `ZeroFillUnit` 可以是 `ReduceUnit` 的退化模式（arity=0），也可以作为独立单元但共享端口仲裁。

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `dst_addr` | 目标输出地址 |

完成事件：

- `node_id`
- `dst_addr`

时序约定：

- `ZeroFill` 固定延迟 1 周期（对齐代码 `node_latency(ZeroFill) = 1`）

### 6.10 WriteBackUnit

职责：

- 将最终输出块写入 `output buffer`

命令载荷：

| 字段 | 含义 |
| ---- | ---- |
| `aux_list_base` | 源块地址列表在 `AuxInputTable` 中的起始位置 |
| `block_count` | 写回块数（即输出行数） |

> **与代码对齐**：当前模拟器中 `WriteBack` 是单一节点，一次性收集所有输出行（`deps = output_node_ids`，`transfer_scalars = rows * D`）。`AuxInputTable` 中存放的是各输出行结果节点的输出地址。写回延迟为 $\lceil \mathrm{block\_count} \cdot d / W_{\mathrm{out}} \rceil$ 。

完成事件：

- `node_id`
- `written_count`

仲裁规则：

- 第一阶段默认为单写回端口
- 同周期若存在多条写回请求，由编译器避免；运行时不做复杂乱序合并

### 6.11 LifecycleCtrl

职责：

- 追踪每个临时块和输入块的剩余消费者数
- 在最后消费者完成时发出释放请求

输入：

- 来自 `DispatchCtrl` 的 `node_done`
- 编译期静态消费者计数表

输出：

- 对 `BufferManager` 的 `free_valid/free_addr`

这是当前 `simulate.rs` 中 `remaining_consumers` 与 `remaining_input_consumers` 语义的硬件化承接。

第一阶段需要一张编译期生成的 `ConsumerCountTable`：

| 字段 | 含义 |
| ---- | ---- |
| `buffer_addr` | 被追踪的块地址（temp 或 input） |
| `initial_count` | 该块的消费者总数 |

`LifecycleCtrl` 维护每个被追踪地址的剩余消费者计数，每收到一个包含该地址的完成事件就递减。当计数归零时向 `BufferManager` 发出 `free` 请求。

此外，LifecycleCtrl 需要知道每个完成节点消费了哪些 buffer 地址。第一阶段通过编译期生成的 `NodeConsumeTable` 提供此映射：

| 字段 | 含义 |
| ---- | ---- |
| `node_id` | 节点编号 |
| `consume_list_base` | 该节点消费的地址列表在 `AuxInputTable` 中的起始位置 |
| `consume_count` | 消费的地址数 |

当 LifecycleCtrl 收到 `node_done(node_id)` 事件时，查 `NodeConsumeTable` 找到该节点消费的所有 buffer 地址，逐一递减对应的 `ConsumerCountTable` 条目。

## 7. 模块间连接关系

推荐的数据流关系如下：

```text
DispatchCtrl ──cmd──> InputReader ──W──> BufferManager(input)
DispatchCtrl ──cmd──> ParamLoader ──W──> SharedParamStore
DispatchCtrl ──cmd──> BroadcastRouter ──R──> BufferManager(input)
                                      ──W──> BufferManager(temp)
DispatchCtrl ──cmd──> KernelArray ──R──> BufferManager(temp/input)
                                  ──R──> SharedParamStore
                                  ──W──> BufferManager(temp)
DispatchCtrl ──cmd──> ReduceUnit  ──R──> BufferManager(temp)
                                  ──W──> BufferManager(output/temp)
DispatchCtrl ──cmd──> ZeroFillUnit ──W──> BufferManager(output/temp)
DispatchCtrl ──cmd──> WriteBackUnit ──R──> BufferManager(temp/output)
                                    ──W──> BufferManager(output)

LifecycleCtrl ──free──> BufferManager(temp/input)
AddrGen 为各模块提供逻辑地址到物理地址映射（组合逻辑）

各模块 ──evt──> DispatchCtrl（完成事件回传）
各模块 ──evt──> LifecycleCtrl（消费者计数递减）
```

### 7.1 硬件参数配置

编译器与硬件共享的参数集（对齐 `SchedParams`）：

| 参数 | 含义 | 第一阶段默认值 |
| ---- | ---- | -------------- |
| `P` | 核阵列并行槽位数 | 4 |
| `B_ports` | 广播端口数 | 2 |
| `R_ports` | 归约端口数（ZeroFill 共享） | 2 |
| `L_core` | 一般核执行延迟 | 4 |
| `L_scale` | 标量核执行延迟 | 1 |
| `direct_latency` | 直通路径延迟 | 0 |
| `B_w` | 参数总线宽度（标量/周期） | 2 |
| `W_out` | 输入/输出搬运带宽（标量/周期） | 4 |
| `bcast_model` | 广播延迟模型 | Tree |
| `reduce_model` | 归约延迟模型 | Tree |

第一阶段这些参数可在 RTL 中固化为常量。若后续需要运行时可配（如支持多种规模），可升级为配置寄存器组。编译器在生成调度表前读取同一参数集，确保编译期假设与硬件一致。

### 7.2 置换快速路径

当 BlockMat 为纯置换矩阵（每行每列恰好一个 `Id` 核，其余为零）时，当前模拟器跳过 `InputRead` 和 `WriteBack`，直接在缓冲区内按置换映射重定向地址（详见 `lower.rs` 中的 `is_permutation` 分支）。

对硬件而言，这意味着：

1. `DispatchCtrl` 需识别调度表中的"置换模式"标记（可在 `BatchTable` 头部增加 `mode` 字段）
2. 置换执行退化为 `AddrGen` 的地址重映射，无数据搬运
3. 编译器在识别 R8 纯置换时，输出精简的调度表（仅含地址映射，无 InputRead / Core / WriteBack 命令）

## 8. 第一阶段仲裁与停顿规则

为了尽快从当前抽象模拟器推进到可写 RTL 的接口级实现，第一阶段建议采用以下保守策略：

1. **调度冲突由编译器主导消解**
   运行时不主动重排，只检查模块 ready。

2. **模块反压触发全局停顿**
   若当前批中任一命令目标模块不可接收，则 `DispatchCtrl` 暂停该批剩余命令发射。

3. **端口内顺序执行**
   同一广播端口、归约端口、写回端口保持 FIFO 顺序，不实现同端口乱序完成。

4. **完成事件可乱序到达，但必须带 `node_id`**
   因为不同模块延迟不同，`DispatchCtrl` 依赖 `node_id` 做 trace 与状态回写。

5. **不做运行时依赖重算**
   第一阶段所有依赖满足性由离线调度保证；运行时只负责执行和停顿。

6. **零延迟级联**
   当某条命令的延迟为 0（如 `DIRECT`、部分零延迟操作），其完成事件在同一调度周期内可见。当前模拟器在同一周期内按拓扑序处理所有完成节点，允许 0 延迟节点的输出在同周期被后续节点消费。RTL 实现中，若不支持组合级联，可统一将最小延迟设为 1 周期，并在编译器端相应调整调度表。

## 9. 与当前软件原型的映射

当前 `csimulator` 的三层含义分别是：

- [csimulator/src/lower.rs](../csimulator/src/lower.rs)：生成抽象调度图节点
- [csimulator/src/schedule.rs](../csimulator/src/schedule.rs)：生成按资源约束发射的时间表
- [csimulator/src/simulate.rs](../csimulator/src/simulate.rs)：按节点完成事件解释数据语义

本文档的作用，就是把上述“抽象节点语义”继续落到“模块接口语义”：

| 当前软件抽象 | 本文档中的硬件/软实现对应 | `resource_class` | 端口限制 |
| ------------ | ------------------------- | ---------------- | -------- |
| `NodeType::InputRead` | `InputReader` | `input` | 1 |
| `NodeType::ParamLoad` | `ParamLoader` | `load` | 1 |
| `NodeType::Broadcast` | `BroadcastRouter` | `bcast` | `B_ports` |
| `NodeType::Core(General/Scalar)` | `KernelArray` | `core` | `P` |
| `NodeType::Direct` | `KernelArray` 直通路径 | `none` | 无限制 |
| `NodeType::Reduce` | `ReduceUnit` | `reduce` | `R_ports` |
| `NodeType::ZeroFill` | `ZeroFillUnit` | `reduce`（共享） | `R_ports` |
| `NodeType::WriteBack` | `WriteBackUnit` | `writeback` | 1 |
| `remaining_consumers` | `LifecycleCtrl` | — | — |

## 10. 推荐落地顺序

如果下一步要从当前原型继续推进，推荐顺序是：

1. 先做一版 **模块级软实现模拟器**
   把 `simulate.rs` 重构为上述模块的逐周期 `tick()` 组合，但仍运行在 Rust 中。

2. 再冻结 `NodeTable` / `BatchTable` 编码
   使编译器输出与模块级软实现共享同一调度表格式。

3. 最后再进入 RTL
   RTL 版本直接复用相同模块边界、接口名称和命令语义。

这样可以避免“软件模拟器语义”和“RTL 接口语义”各写一套，后面再艰难对齐。

## 11. 本版结论

截至本文档，第一阶段从 BlockMat 到调度图的语义已经清楚；缺的不是模块职责，而是 **模块接口、命令格式、地址/句柄编码、仲裁规则和停顿语义**。

本文档给出的第一版答案是：

- 模块采用 `DispatchCtrl + BufferManager + 执行模块 + LifecycleCtrl` 的组织方式
- 命令与事件统一采用 `valid/ready` 风格接口
- 运行时以编译器生成的 `BatchTable` / `NodeTable` 为主控制面
- 第一阶段默认静态调度、全局停顿、端口内 FIFO、完成事件带 `node_id`

这已经足够作为“模块级软实现”和“RTL 前置规格”的起点，但还不是最终投产规格。
